# ステージ4：Deliver — 仮説検証 + プロダクトスペックサマリー

## 4.5 仮説検証プラン

**対象：中/高完成度 / オーディエンスがデータサイエンティスト/自分自身の場合**

```
| コア仮説 | 検証方法 | 検証タイミング | 成功基準 | 仮説が誤りの場合... |
|----------------|-------------------|-------------------|-----------------|----------------------------------|
| | Alphaユーザーインタビュー / データ | | | |
```

原則：定性的検証（ユーザーインタビュー、日記調査）+ 定量的検証（データ）は同等の重み；自身の確証バイアスに警戒を。

## 4.6 プロダクトスペックサマリー

プロダクトスペックサマリーは、すべてのモードの最終成果物です（クイックモードは1ページの方向性サマリーを作成 — 構造は異なるが同じ精神）。
プロダクトスペックサマリーは3セクションで構成され、異なる読み深さに対応：

### セクション1：意思決定サマリー（30秒で読める）

```
**プロダクトワンライナー**：[PR-FAQ見出し]
**誰のため**：[ターゲットペルソナを一文で]
**コア価値**：[JTBD機能的ジョブ + 感情的ジョブを一文で]
**North Star Metric**：[メトリクス名 + 定義]
**現在のPMFレベル**：[レベル1-4 + 一文の説明]
```

### セクション2：実行境界（2分で読める）

```
**コアJTBD Statement**：[ターゲット顧客] + [ジョブコンテキスト]で + [ジョブ]をしたい
**Aha Moment**：ユーザーが[アクション]を完了した時、コア価値を体験。目標：[X]以内に到達。
**MVP必須機能**：[最も重要な3-5の機能/能力]
**やらないこと（このバージョンで意図的に除外）**：[明示的な除外 + 理由]
**主要仮説**：[検証が必要な2-3のコア仮説]
**Pre-mortemトップリスク**：[失敗の最も可能性の高い2-3の原因]
**Hero Metric**：[最も重要な1-2の成功メトリクス + 目標値]
```

### セクション3：ディープリファレンス（必要時に参照）

```
**JTBD分解**：
  - 機能的ジョブ：
  - 感情的ジョブ：
  - 社会的ジョブ：
**April Dunfordポジショニング**：
  - 実際の競合代替品：
  - ユニーク属性：
  - ターゲットユーザーへのコア価値：
  - ターゲット市場特性：
  - 市場カテゴリー：
**コアHMW質問**：[Defineステージから]
**ソリューション説明**：[PR-FAQサマリーから]
**ビジネスモデル**：[収益モデル + プライシング戦略（完了している場合）]
**GTM戦略**：[初期獲得チャネル + ローンチ戦略（完了している場合）]
```

### 🔍 ギャップと盲点

Claudeは会話フロー全体に基づいて、以下のタイプのギャップを積極的に特定：

```
| タイプ | 説明 | 推奨次ステップ |
|------|------------|----------------------|
| 未実行ステップ | [どのフレームワークがスキップされ、どんな盲点が生じる可能性があるか] | |
| データ不足 | [どの意思決定が実際のユーザーデータではなく仮定に基づいているか] | |
| 論理の飛躍 | [推論のどこに中間的なエビデンスが欠けているか] | |
| 未検証の仮説 | [まだ検証方法のないすべてのコア仮説をリスト] | |
| 不明確な定義 | [実行前により正確な定義が必要な概念] | |
```

### 💡 追加推奨事項

Claudeはプロダクトの特性、ステージ、既知のリスクに基づいて3-5の具体的でアクション可能な推奨事項を提供：

```

2. [次の優先]：[具体的なアクション] — 理由：

...
```
