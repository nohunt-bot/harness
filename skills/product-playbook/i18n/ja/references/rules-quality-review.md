# 🔍 品質レビュールール

> 各ステップ完了時に読み込みます。本ファイルは「レビュープロセス」と「チェック基準」を分離して管理し、品質セルフチェックメカニズムが確実に機能するようにします。

## ⚙️ レビュープロセス（全モード共通）

各ステップの成果物を出力した後、Claude は以下のレビュープロセスを実行する必要があります：

### ステップ1：項目ごとのセルフチェック

1. 該当ステップに対応する品質チェックリストを確認（下記「チェック基準」セクション参照）
2. 各項目を ✅ または ❌ でマーク
3. ❌ の項目には「どう改善するか」の具体的な説明を付ける

### ステップ2：強制的な批判

- **すべて ✅ にしてはならない**：すべての項目が合格した場合、「この成果物の最も弱い点」を主体的に指摘し、強化方法を説明する
- これは重箱の隅をつつくことではなく、セルフレビューがゴム印にならないようにするため
- Amazon PR-FAQ の精神を参考に：品質とは問題を見つけることであり、問題がないことを確認することではない

### ステップ3：表示フォーマット

```
📝 品質セルフチェック：
- ✅ [チェック項目]
- ✅ [チェック項目]
- ❌ [チェック項目] → 改善方向：[具体的な説明]
⚠️ 最も弱い点：[説明] → 強化提案：[具体的なアクション]
```

## 📋 チェック基準（ステップ別）

> 以下は各フレームワーク成果物の独立したチェックリストです。レビュープロセスは変わりません。対応するチェックリストに差し替えるだけです。

### Persona（ペルソナ）

| # | チェック項目 |
|---|------------|
| 1 | セグメントは「用途/タスク/動機」に基づいているか？（人口統計ではなく） |
| 2 | 各 Persona は MECE か？（相互に排他的で、ターゲット市場を完全にカバー） |
| 3 | コア TA と二次 TA が明確に区別されているか？ |
| 4 | 各 Persona の「課題/チャレンジ」は実際の観察または合理的な推論に基づいているか？ |
| 5 | 「現在の対処法とその理由」はワークアラウンドを特定できるほど具体的か？ |

よくある問題：年齢・性別でセグメント分け、Persona 間の差異が不明瞭、ペインポイントが漠然としている

### JTBD（Jobs to Be Done）

| # | チェック項目 |
|---|------------|
| 1 | 具体的な状況が含まれているか？（「いつでもどこでも」ではなく「深夜に銀行に連絡できない時」） |
| 2 | 単一のコアジョブにフォーカスしているか？（3つのジョブを1文にまとめていないか） |
| 3 | 機能的/感情的/社会的ジョブがすべて識別されているか？ |
| 4 | 「このソリューションがこのジョブを解決するか」を判断するために使えるか？ |
| 5 | 「現在のやり方」と「ギャップ」が含まれているか？（ギャップ = 機会） |
| 6 | 5つの深掘り質問の Q5 は感情的動機/職業的アイデンティティ/心理的恐怖に触れているか？（機能的な記述ではなく） |

### Positioning（ポジショニング）

| # | チェック項目 |
|---|------------|
| 1 | 「競合代替品」はユーザー視点からか？（ユーザーが実際に代わりに使うもの、あなたが考える競合ではなく） |
| 2 | 「独自の属性」は競合代替品ができない、またはうまくできないことか？ |
| 3 | 「ユーザーにとっての価値」はユーザーの言葉か、プロダクトの言葉か？（「2時間節約」vs「AI駆動の自動化」） |
| 4 | 「ターゲット市場」はその人たちを見つけられるほど具体的か？ |
| 5 | 5つのポジショニング要素間に論理的一貫性があるか？ |

よくある問題：独自の属性と価値の間の断絶、マーケットカテゴリの誤りにより誤った基準で評価される

### HMW（How Might We）

| # | チェック項目 |
|---|------------|
| 1 | 明確な制約条件があるか？（完全にオープンエンドではなく） |
| 2 | 十分なソリューション空間が残されているか？（唯一の解を指していないか） |
| 3 | 1つの JTBD またはペインポイントに直接マッピングできるか？ |
| 4 | チームがこの HMW を見てソリューションの発想を始められるか？ |

よくある問題：大きすぎ（ビジョンの言い換え）、小さすぎ（ソリューションの指定）、複数の問題が混在

### PR-FAQ（プレスリリース + FAQ）

| # | チェック項目 |
|---|------------|
| 1 | 見出しはユーザー視点か？（「ユーザーが X できるようになる」vs「Y 機能をリリース」） |
| 2 | 第1段落で読者が10秒以内に「なぜこれが重要か」を理解できるか？ |
| 3 | ペインポイントの記述は実際のユーザーシナリオからか？ |
| 4 | ソリューション段落の最初の文はユーザーの体感/シナリオで始まるか？（機能的な動詞ではなく） |
| 5 | ユーザー引用は本物の人が言いそうな言葉か？ |
| 6 | FAQ には既存ツールとの比較に関する鋭い質問が含まれているか？ |

よくある問題：見出しがニュースではなくプロダクト告知のように読める、ソリューション段落が機能リストになる、FAQ が答えやすい質問ばかり

### North Star Metric（北極星指標）

| # | チェック項目 |
|---|------------|
| 1 | ユーザーが得る価値を反映しているか？（売上やDAUではなく） |
| 2 | 継続的に成長できるか？（自然に上限に達しないか） |
| 3 | チーム全員がこの指標を見て何をすべきか分かるか？ |
| 4 | 操作可能か？（可能な場合、ガードレール指標が必要） |
| 5 | B2B プロダクト：個人ユーザーだけでなく、顧客組織レベルの価値を反映しているか？ |

よくある問題：売上を North Star にする（売上は結果であり駆動因子ではない）、指標が複合的すぎてアクションにつながらない

### Aha Moment（アハモーメント）

| # | チェック項目 |
|---|------------|
| 1 | 具体的で追跡可能な行動か？（「プロダクトが使いやすいと感じる」ではなく） |
| 2 | JTBD の機能的ジョブと直接関連しているか？ |
| 3 | 到達時間の目標は合理的か？（B2C は初回使用内、B2B はトライアル期間内） |
| 4 | ユーザーをより早く到達させるオンボーディングを設計できるか？ |

### セキュリティチェック

> 詳細な基準は `references/08-security-checklist.md` を参照。以下は品質セルフチェックの要約：

| # | チェック項目 |
|---|------------|
| 1 | 認証方式が明確に選定済み、「未定」ではない |
| 2 | 少なくとも3つのセキュリティヘッダーが計画済み |
| 3 | Rate Limiting 戦略がプロダクト特性に合わせて調整済み（テンプレートのコピーではなく） |
| 4 | .gitignore にすべての機密ファイルが含まれている |

### ドキュメントエクスポート

> 詳細な基準は `references/rules-export-document.md` を参照。

| # | チェック項目 |
|---|------------|
| 1 | HTML に Markdown 構文が残っていない（`**`、`##`、`|---|`） |
| 2 | すべてのテーブルの行数と列数が元のコンテンツと一致している |

## 🔄 ステップ間一貫性レビュー

> フロー終了時に読み込み。詳細なモード別チェック項目は `references/rules-end-of-flow.md` を参照。

ステップ間一貫性レビューは、単一ステップの品質セルフチェックとは独立した第2層のレビューで、すべてのステップ完了後に実行されます。その目的は、反復プロセス中に上流の修正が引き起こした下流の不整合を捕捉することです。

### クイックチェック項目（全モード共通）

| # | チェック次元 | 検証質問 |
|---|------------|---------|
| 1 | ターゲットユーザーの一貫性 | JTBD、ポジショニング、PR-FAQ は同じユーザー群を指しているか？ |
| 2 | コア課題の一貫性 | PR-FAQ は JTBD で述べた課題に対応しているか？MVP は解決しているか？ |
| 3 | ソリューション ↔ スコープ | 選択したソリューションは MVP スコープと一致しているか？ |
| 4 | 指標 ↔ 価値 | North Star は JTBD の成果を測定しているか？ |
| 5 | リスクの適時性 | Pre-mortem のリスクは最終ソリューションにまだ関連しているか？ |

## 📐 設計原則

本ファイルは Reviewer デザインパターンに従います：**チェック基準とレビュープロセスを分離して管理**。

- **プロセスは不変**：ステップ1（項目ごとのセルフチェック）→ ステップ2（強制的な批判）→ ステップ3（表示フォーマット）
- **基準は差し替え可能**：新しいフレームワーク追加時は、「チェック基準」セクションに対応するチェックリストを追加するだけ
- **独立した読み込み**：各リファレンスファイルにはインラインリマインダーとして埋め込みチェックリストが残されており、本ファイルは統一レビューエントリーポイントとして機能
