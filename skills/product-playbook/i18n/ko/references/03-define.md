## 2.2 April Dunford의 포지셔닝 프레임워크

**적용: 중간/높은 완성도 / 독자가 경영진/영업/마케팅인 경우**

포지셔닝은 태그라인이 아닙니다 — 어디서 경쟁하고 누구를 위한 것인지를 결정하는 것입니다:

```
| 포지셔닝 요소 | 질문 | 답변 |
|-------------|------|------|
| 경쟁 대안 | 당신의 제품이 없다면 사용자는 무엇을 사용할 것인가? (진짜 답, 당신이 경쟁자라고 생각하는 것이 아닌) | |
| 고유 속성 | 경쟁 대안에 없는 것이 무엇인가? | |
| 사용자 가치 | 이 고유 속성이 사용자에게 어떤 실질적 가치를 제공하는가? | |
| 타겟 시장 특성 | 이 가치를 가장 중요시하는 사용자는 누구인가? (구체적일수록 좋음) | |
| 시장 카테고리 | 당신의 제품을 가장 잘 포지셔닝하는 시장 프레임은? (프레임이 경쟁 기준을 결정) | |
```

가장 흔한 포지셔닝 실수:
- 기능을 포지셔닝으로 취급 ("우리는 AI가 있어요!"는 포지셔닝이 아닙니다)
- 너무 넓게 포지셔닝 — 모든 사람을 커버하면 아무도 커버하지 못함
- 사용자의 실제 대안이 아닌 자신이 인식하는 경쟁자를 기준으로 삼기

### 📝 포지셔닝 품질 체크리스트
- ✅ "경쟁 대안"이 사용자 관점인가? (당신이 경쟁자라고 생각하는 것이 아닌, 사용자가 실제로 대신 사용하는 것)
- ✅ "고유 속성"이 경쟁 대안이 할 수 없거나 잘 할 수 없는 것인가? (둘 다 가지고 있는 것이 아닌)
- ✅ "사용자 가치"가 사용자 언어로 표현되었는가 아니면 제품 언어인가? ("2시간 절약" vs. "AI 기반 자동화")
- ✅ "타겟 시장"이 실제로 이 사람들을 찾을 수 있을 만큼 구체적인가?
- ✅ 5개 포지셔닝 요소가 논리적으로 일관성이 있는가?
- ❌ 흔한 문제: 고유 속성과 가치 간 단절, 잘못된 시장 카테고리로 인해 잘못된 기준으로 평가받음

## 2.3 HMW (How Might We) 문제 재구성

페인포인트를 HMW 질문으로 변환하고, JTBD 렌즈를 결합하여 각 HMW 뒤의 Job 유형을 확인하세요:

```
| 페인포인트 # | 페인포인트 | 대응 JTBD 유형 | HMW 질문 |
|-------------|----------|---------------|---------|
| P1 | [페인포인트 설명] | 기능적 / 감정적 / 사회적 | 어떻게 하면... |
| P2 | [페인포인트 설명] | | 어떻게 하면... |
```

HMW 세분화 원칙:
- 너무 넓음 ("사용자를 더 행복하게 만드는 방법") → 방향성 없음
- 적절함 ("사용자가 최초 설정을 60초 내에 완료하게 하는 방법") → 제약이 있되 개방적
- 너무 좁음 ("버튼 색상을 바꾸는 방법") → 가능성 제한

### 📝 HMW 품질 체크리스트
- ✅ 명확한 제약이 있는가? (완전히 열린 질문이 아닌)
- ✅ 여러 솔루션의 여지를 남기는가? (단일 답을 가리키지 않는)
- ✅ JTBD 또는 페인포인트에 직접 매핑할 수 있는가?
- ✅ 팀이 이 HMW를 보고 솔루션 브레인스토밍을 시작할 수 있는가?
- ❌ 흔한 문제: 너무 넓음 (비전 재언급), 너무 좁음 (솔루션 명시), 여러 문제 혼합

**예시:**
- ❌ 너무 넓음: "어떻게 하면 사용자를 더 만족시킬 수 있을까?"
- ✅ 적절함: "어떻게 하면 첫 주택 구매자가 3분 내에 감당 가능한 모기지 금액을 계산하게 할 수 있을까?"
- ❌ 너무 좁음: "어떻게 하면 홈페이지에 모기지 계산기를 추가할 수 있을까?"

## 2.4 기회 평가 테이블

HMW 질문의 우선순위를 정하세요:

```
| HMW 질문 | 영향받는 Persona | Persona 규모 | 사용자 임팩트 (1-5) | 비즈니스 가치 (1-5) | 실현 가능성 (1-5) | 합계 | 우선순위 |
|---------|----------------|-------------|-------------------|-------------------|-----------------|------|---------|
| | [영향받는 Persona 나열] | [대/중/소] | | | | | |
```

**점수 척도 정의:**

| 점수 | 사용자 임팩트 | 비즈니스 가치 | 실현 가능성 |
|------|-------------|-------------|------------|
| 1 | 소수 사용자에게 가벼운 불편 | 간접적, 기껏해야 장기적 효과 | 완전히 새로운 기술 또는 광범위한 R&D 필요 |
| 2 | 일부 사용자가 가끔 겪음 | 일부 지표에 간접적 영향 가능 | 상당한 새로운 역량 구축 필요 (3개월 이상) |
| 3 | 핵심 TA가 정기적으로 겪음 | 핵심 지표에 긍정적 영향 | 일부 새로운 개발이 필요하지만 기술적으로 가능 (1-3개월) |
| 4 | 많은 사용자가 빈번히 겪음 | 사용자 성장 또는 리텐션을 직접 견인 | 현재 팀 역량 범위, 2-4주 |
| 5 | 많은 사용자가 매일 핵심 과업을 완수하지 못함 | 매출을 직접 견인하거나 North Star Metric에 중대한 영향 | 현재 팀이 2주 내에 완료 가능 |

**Shreyas Doshi의 기회비용 사고:**

"이 기능의 ROI가 얼마인가?"라고 묻지 마세요. 대신 질문하세요:

> "A에 리소스를 투자하면, B에 투자할 기회를 포기하게 됩니다. A가 B보다 가치 있다고 확신하는가?"

ROI 사고는 단일 기회의 가치를 평가합니다; 기회비용 사고는 모든 기회 중에서 더 나은 선택을 하도록 도와줍니다.

**0-to-1 집중 리마인더:** 기회 평가를 완료한 후, MVP 핵심으로 **최우선 HMW 질문 단 하나만** 선택하는 것을 권장합니다. (Facebook: 대학생 → 고등학생 → 모든 사람; 프로필 페이지 → 사진 → 뉴스 피드)
