# 제품 전략 계층

> **참고: 각 모드에서 실행할 단계는 SKILL.md의 "모드 단계 시퀀스"가 권위 있는 기준입니다. 아래 내용은 참고용입니다.**

**적용: 풀 모드, 높은 완성도, 독자가 경영진/크로스펑셔널 정렬인 경우**
**생략 가능: 퀵 모드, 빌드 직행 모드, 낮은 완성도**

## Strategy Blocks (Chandra Janakiraman / Headspace / Meta)

좋은 전략의 계층 구조 — 각 층은 다음 층의 기반입니다:

```
Mission
  └→ Vision — 5-10년 후 세상이 어떤 모습이길 원하는가?
       └→ Strategy — 그 비전에 어떻게 도달할 것인가? (핵심 선택과 트레이드오프)
            └→ Goals / OKRs — 향후 6-12개월의 우선순위
                 └→ Roadmap — 구체적으로 무엇을 만들 것인가?
                      └→ Tasks — 누가, 언제, 무엇을 하는가?
```

## Richard Rumelt의 좋은 전략의 핵심

- **진단(Diagnosis)**: 직면한 도전을 명확히 정의 (모든 문제가 아닌 가장 중요한 문제)
- **방침(Guiding Policy)**: 전반적 접근 방식 (목표가 아닌 방법)
- **일관된 행동(Coherent Actions)**: 서로 강화하는 구체적 행동, 독립적인 계획의 집합이 아님

> 나쁜 전략의 징후: 진단 없는 거창한 목표; 공허한 사고를 가리는 화려한 언어; 모든 계획을 "전략"이라고 부르기.

## Shreyas Doshi의 제품 업무 3단계

제품 문제를 다루기 전에, 어떤 수준에서 작업하고 있는지 파악하세요:

```
Level 3: 제품 탁월성 — 올바른 일을 탁월하게 잘 하기
Level 2: 제품 전략 — 올바른 일을 하기
Level 1: 제품 기반 — 일을 할 수 있는 기반 갖추기 (문화, 프로세스, 인재)
```

> 대부분의 PM은 Level 2 문제를 방치하면서 Level 3에 너무 많은 시간을 씁니다. 대부분의 이른바 "실행 문제"는 근본적으로 전략 문제입니다.

## Shreyas Doshi의 LNO 시간 배분 프레임워크

매주 작업 항목에 대해 먼저 질문하세요: 이것이 제품에 어떤 유형의 임팩트를 주는가?

```
L (Leverage): 전략, 비전, 문화 → 충분한 시간을 투자하고 탁월함을 추구
N (Neutral): 일반적인 협업, 루틴 커뮤니케이션 → 잘 하되 완벽을 추구하지 않기
O (Overhead): 행정, 회의, 서류 작업 → 빠르게 끝내고 과잉 투자하지 않기
```

> 절약한 O 시간을 방치된 L 업무에 재투자하세요.

## OKR 작성 가이드라인

Strategy Blocks의 Goals/OKRs는 전략을 하위로 전파하는 핵심 계층입니다. 좋은 OKR을 작성하기 위한 최소 규칙:

**Objective**: 정성적이고, 영감을 주며, 이해하기 쉬운. 달성하고 싶은 상태를 설명하지, 할 일 항목이 아닙니다.
- ✅ 좋은 O: "신규 사용자가 첫날 제품의 핵심 가치를 느끼게 하기"
- ❌ 나쁜 O: "온보딩 재설계 완료" (그것은 작업이지, 목표가 아닙니다)

**Key Results**: 정량적이고, 측정 가능하며, 기한이 있는. 목표 달성 여부를 어떻게 알 수 있는지 설명합니다.
- ✅ 좋은 KR: "신규 사용자 D1 핵심 행동 완료율을 20%에서 40%로 증가"
- ❌ 나쁜 KR: "새로운 온보딩 플로우 출시" (그것은 산출물이지, 성과가 아닙니다)

**흔한 함정:**
- 작업 목록을 OKR로 위장 ("기능 X 완료"는 KR이 아닙니다)
- OKR이 너무 많음 (분기당 Objective 2-3개, Objective당 KR 3-5개가 목표)
- 서로 모순되거나 무관한 KR
- 후행 지표만 있고 선행 지표가 없음

**예시:**
```
O: 타겟 사용자가 우리 제품을 사랑하게 만들기 (PMF Level 2 → Level 3)
  KR1: D28 리텐션을 12%에서 20%로
  KR2: Sean Ellis Score를 28%에서 40%로
  KR3: 월간 자발적 추천 비율을 10%에서 25%로
```

## 세 가지 핵심 제품 질문 (전체 프로세스에 걸쳐)

이 세 가지 질문은 순서대로 답해야 합니다 — **순서를 바꿀 수 없습니다**:

> **Q1: 사용자를 어떻게 문 앞으로 데려올 것인가?**
> **Q2: Aha Moment에 어떻게 가장 빠르게 도달하게 할 것인가?**
> **Q3: 핵심 가치를 어떻게 반복적으로 전달할 것인가?**

Define 단계에서 이것은 다음으로 변환됩니다:
- **누구를 위한 것인가?**
- **왜 만드는가?**
- **그것은 무엇인가?**
