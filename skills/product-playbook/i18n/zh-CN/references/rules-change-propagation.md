# 🔄 变更传播规则（Change Propagation）

> 当使用者修改已完成步骤时载入。

## 📍 进度指示器（每个步骤都必须显示）

在执行任何步骤时，Claude 必须在回应的最开头显示进度列，格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 [执行模式] ｜ 进度 S[目前步骤编号] / S[总步骤数]
✅ S1：[步骤名称]（已完成）
✅ S2：[步骤名称]（已完成）
▶️ S3：[步骤名称]（进行中）
⬜ S4：[步骤名称]（待执行）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

此进度指示器在以下情况都必须出现：
- 每进入一个新步骤
- 使用者回到某个步骤进行修改时
- 完成某个步骤、询问使用者确认后进入下一步时

## 触发方式
- 「回到 Persona」「回到 JTBD」「回到 HMW」「回到 PR-FAQ」等任意步骤名称
- 「我想修改一下 [步骤名]」「[步骤名] 的部分我想改」
- 直接对某个已产出的表格或内容说「这里改成...」

## 修改后的必要动作

当任何步骤被修改，Claude **必须主动执行以下检查**：

```
修改层级               受影响的下游（必须重新确认或更新）
─────────────────────────────────────────────────────
Persona / JTBD        → HMW、机会评估表、Positioning、PR-FAQ、North Star、产品规格摘要
HMW / 机会评估        → PR-FAQ、平行解法、MVP、North Star、产品规格摘要
Positioning           → PR-FAQ、产品规格摘要
PR-FAQ / 解法         → Pre-mortem、GEM/RICE、MVP、Aha Moment、产品规格摘要
MVP / Not Doing List  → User Story、DB schema（若已产出）、产品规格摘要
North Star / 指标     → 假设验证计划、产品规格摘要
产品规格摘要           → HTML 报告、PRD（若已产出）
```

### 功能扩充依赖关系：
```
功能扩充依赖关系：
─────────────────────────────────────────────────────
S1（问题 + 情境）     → S2（解法）、S3（风险）、S4（执行范围）
S2（选定解法）        → S3（风险）、S4（执行范围）
S3（风险评估）        → S4（执行范围）
```

## 执行方式

1. **告知使用者影响范围**：「你修改了 [步骤]，这会影响到 [下游步骤清单]，我会逐一更新。」
2. **逐一确认或自动更新下游**：
   - 若下游改动小（措辞调整）→ 直接更新，说明改了什么
   - 若下游改动大（方向转变）→ 提示使用者确认新方向再更新
3. **最终重新整合产品规格摘要**
4. **若已产出 HTML 报告或 PRD**：直接重新产出，并输出版本快照：

```
📋 版本快照 v[旧版号] → v[新版号]
修改的步骤：[步骤名称]
修改前的关键内容：[1-3 句话]
修改后的关键内容：[1-3 句话]
连带更新的下游：[哪些步骤被一并调整]
```

## 原则
- 任何修改都不会静默发生——必须明确告知影响范围
- 使用者有权选择「只改这一步，下游暂时不动」，Claude 需标记哪些部分已过时（加上 ⚠️ 待更新 标记）
- 修改历史在对话中保持可追溯
