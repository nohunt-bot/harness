## 2.2 April Dunford 的产品定位框架（Positioning）

**适用于：中/高完整性 / 产出对象为老板/业务/行销**

定位不是口号，而是决定你在哪里竞争、为谁竞争：

```
| 定位要素 | 问题 | 你的答案 |
|---------|------|---------|
| 竞争替代品 | 如果你的产品不存在，用户会用什么？（真实答案，不是你认为的竞争对手） | |
| 独特属性 | 你有哪些是竞争替代品没有的功能或特性？ | |
| 对用户的价值 | 这些独特属性对用户带来什么实际价值？ | |
| 目标市场特征 | 哪些用户最在意这些价值？（越具体越好） | |
| 市场类别 | 你的产品最好被放在哪个市场框架下看待？（框架决定竞争标准） | |
```

最常见的定位错误：
- 把功能当成定位（「我们有 AI！」不是定位）
- 定位太模糊，覆盖所有人等于不覆盖任何人
- 以自己认为的竞争对手为参照，而不是用户实际的替代方案

### 📝 Positioning 品质自检清单
- ✅ 「竞争替代品」是否来自用户视角？（用户真的会用什么替代，不是你认为的竞对）
- ✅ 「独特属性」是否是竞争替代品做不到或做不好的？（不是你有别人也有的东西）
- ✅ 「对用户的价值」是用户语言还是产品语言？（「节省 2 小时」vs「AI 驱动自动化」）
- ✅ 「目标市场」是否够具体到能找到这些人？
- ✅ 五个定位要素之间是否逻辑一致？
- ❌ 常见问题：独特属性和价值之间断裂、市场类别选错导致被错误标准评判

## 2.3 HMW（How Might We）问题转化

将痛点转化为 HMW 问题，同时结合 JTBD 视角确认每个 HMW 背后对应的工作类型：

```
| 痛点编号 | 痛点 | 对应 JTBD 类型 | HMW 问题 |
|---|---|---|---|
| P1 | [痛点描述] | 功能性 / 情感性 / 社交性 | 我们可以如何... |
| P2 | [痛点描述] | | 我们可以如何... |
```

HMW 粒度原则：
- 太大（「如何让用户更开心」）→ 没有方向感
- 刚好（「如何让用户在 60 秒内完成第一次设定」）→ 有约束又有空间
- 太小（「如何改按钮颜色」）→ 限制了可能性

### 📝 HMW 品质自检清单
- ✅ 是否有明确的约束条件？（不是完全开放式）
- ✅ 是否保留了足够的解法空间？（不是指向唯一解）
- ✅ 是否能直接对应到一个 JTBD 或痛点？
- ✅ 团队看到这个 HMW 是否能开始发想解法？
- ❌ 常见问题：太大（等于重述愿景）、太小（等于指定解法）、多个问题混在一起

**范例：**
- ❌ 太大：「我们可以如何让用户更满意？」
- ✅ 刚好：「我们可以如何让首购族在 3 分钟内算出自己负担得起的房贷金额？」
- ❌ 太小：「我们可以如何在首页加一个房贷计算机？」

## 2.4 机会评估表

对 HMW 问题进行优先排序：

```
| HMW 问题 | 影响 Persona | Persona 规模 | 用户影响（1-5） | 商业价值（1-5） | 可行性（1-5） | 总分 | 优先级 |
|---|---|---|---|---|---|---|---|
| | [列出受影响的 Persona] | [大/中/小] | | | | | |
```

**评分量尺定义：**

| 分数 | 用户影响 | 商业价值 | 可行性 |
|------|---------|---------|--------|
| 1 | 少数人偶尔不便 | 间接且长期才可能有回报 | 需要全新技术或大量未知研究 |
| 2 | 部分用户偶尔遇到 | 可能间接带动部分指标 | 需要显著的新能力建设（3+ 个月） |
| 3 | 核心 TA 定期遇到 | 对关键指标有正面影响 | 需要一些新开发但技术上可行（1-3 个月） |
| 4 | 大量用户频繁遇到 | 直接推动用户增长或留存 | 现有团队能力范围，需 2-4 周 |
| 5 | 大量用户每天无法完成核心任务 | 直接驱动营收或大幅影响北极星指标 | 现有团队两周内可完成 |

**Shreyas Doshi 的机会成本思维：**

不要问「这个功能的 ROI 是多少？」，而是问：

> 「如果我把资源投入 A，我放弃了投入 B 的机会。我确定 A 比 B 更值得吗？」

ROI 思维让你评估单个机会是否值得；机会成本思维让你在所有机会之间做出更好的选择。

**0-to-1 聚焦提醒：** 完成机会评估后，建议只选**一个最高优先级的 HMW 问题**作为 MVP 核心。（Facebook：大学生→高中生→所有人；个人主页→相片→动态墙）
