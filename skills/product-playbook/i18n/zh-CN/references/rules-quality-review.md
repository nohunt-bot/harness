# 🔍 品质审查规则

> 每个步骤完成时载入。本文件将「审查流程」与「检查标准」分开维护，确保品质自检机制真正运作。

## ⚙️ 审查流程（所有模式通用）

每个步骤产出后，Claude 必须执行以下审查流程：

### 步骤一：逐项自检

1. 查找该步骤对应的品质自检清单（见下方「检查标准」段落）
2. 逐项标记 ✅ 或 ❌
3. ❌ 项目必须附上「如何改善」的具体说明

### 步骤二：强制批判

- **不得全部 ✅**：如果所有项目都通过，必须主动指出「这份产出最弱的一个环节」并说明如何补强
- 这不是刻意找碴，而是确保自我审查不沦为橡皮图章
- 参考 Amazon PR-FAQ 精神：品质是靠找到问题，不是确认没问题

### 步骤三：呈现格式

```
📝 品质自检：
- ✅ [检查项目]
- ✅ [检查项目]
- ❌ [检查项目] → 改善方向：[具体说明]
⚠️ 最弱环节：[说明] → 补强建议：[具体行动]
```

## 📋 检查标准（依步骤分类）

> 以下为各框架产出的独立检查清单。审查流程不变，只需替换对应的检查清单即可。

### Persona（人物志）

| # | 检查项目 |
|---|---------|
| 1 | 切分是否基于「用途/任务/动机」而非人口统计？ |
| 2 | 各 Persona 之间是否 MECE（互斥且完整覆盖目标市场）？ |
| 3 | 是否明确指出核心 TA vs 次要 TA？ |
| 4 | 每个 Persona 的「问题/挑战」是否来自真实观察或合理推论？ |
| 5 | 「现在做法与理由」是否具体到可以识别 Workaround？ |

常见问题：按年龄性别分群、Persona 之间差异不明显、痛点太笼统

### JTBD（Jobs to Be Done）

| # | 检查项目 |
|---|---------|
| 1 | 是否包含具体情境？（不是「随时随地」，而是「在深夜不方便联络银行时」） |
| 2 | 是否聚焦在一个核心工作？（不是三个 Job 合并成一句） |
| 3 | 功能性/情感性/社交性工作是否都有识别？ |
| 4 | 是否能用来判断「这个解法有没有解决这个 Job」？ |
| 5 | 是否包含「现行做法」和「差距」？（差距 = 机会） |
| 6 | 深挖五问的 Q5 是否触及情感动机/职业认同/心理恐惧？（非功能性描述） |

### Positioning（定位）

| # | 检查项目 |
|---|---------|
| 1 | 「竞争替代品」是否来自用户视角？（用户真的会用什么替代，不是你认为的竞对） |
| 2 | 「独特属性」是否是竞争替代品做不到或做不好的？ |
| 3 | 「对用户的价值」是用户语言还是产品语言？（「节省 2 小时」vs「AI 驱动自动化」） |
| 4 | 「目标市场」是否够具体到能找到这些人？ |
| 5 | 五个定位要素之间是否逻辑一致？ |

常见问题：独特属性和价值之间断裂、市场类别选错导致被错误标准评判

### HMW（How Might We）

| # | 检查项目 |
|---|---------|
| 1 | 是否有明确的约束条件？（不是完全开放式） |
| 2 | 是否保留了足够的解法空间？（不是指向唯一解） |
| 3 | 是否能直接对应到一个 JTBD 或痛点？ |
| 4 | 团队看到这个 HMW 是否能开始发想解法？ |

常见问题：太大（等于重述愿景）、太小（等于指定解法）、多个问题混在一起

### PR-FAQ（新闻稿 + 常见问题）

| # | 检查项目 |
|---|---------|
| 1 | 标题是否从用户角度出发？（「用户能做到 X」vs「我们推出了 Y 功能」） |
| 2 | 第一段是否能让读者在 10 秒内理解「这为什么重要」？ |
| 3 | 痛点描述是否来自真实用户场景？ |
| 4 | 解法段第一句是否以用户感受/场景开头（非功能动词）？ |
| 5 | 用户引言是否像真人会说的话？ |
| 6 | FAQ 是否包含了对比现有工具的尖锐质疑？ |

常见问题：标题像产品公告不像新闻、解法段变成功能列表、FAQ 都是容易回答的问题

### North Star Metric（北极星指标）

| # | 检查项目 |
|---|---------|
| 1 | 是否反映用户获得的价值？（不是营收、不是 DAU） |
| 2 | 是否能持续增长？（不会天然到达上限） |
| 3 | 团队每个人看到这个指标是否都知道自己该做什么？ |
| 4 | 是否可以被操弄？（如果可以，需要护栏指标） |
| 5 | B2B 产品：是否反映客户组织层级的价值，而非只有个人使用者？ |

常见问题：用营收当 North Star（营收是结果不是驱动因素）、指标太复合无法行动

### Aha Moment（顿悟时刻）

| # | 检查项目 |
|---|---------|
| 1 | 是否是一个具体的、可追踪的行为？（不是「感觉产品好用」） |
| 2 | 是否与 JTBD 的功能性工作直接相关？ |
| 3 | 到达时间目标是否合理？（B2C 应在首次使用内；B2B 可能在试用期内） |
| 4 | 是否可以设计 Onboarding 来引导用户更快到达？ |

### 安全性检查

> 详细标准见 `references/08-security-checklist.md`，以下为品质自检摘要：

| # | 检查项目 |
|---|---------|
| 1 | 认证方式已明确选定，不是留空的「待确认」 |
| 2 | 至少 3 个安全性 Headers 已规划 |
| 3 | Rate Limiting 策略已针对产品特性调整（不是直接复制模板） |
| 4 | .gitignore 已包含所有敏感文件 |

### 文件导出

> 详细标准见 `references/rules-export-document.md`。

| # | 检查项目 |
|---|---------|
| 1 | 无残留 Markdown 语法在 HTML 中（`**`、`##`、`|---|`） |
| 2 | 所有表格列数和栏数与原始内容一致 |

## 🔄 跨步骤一致性审查

> 流程结束时载入。详细的模式别检查项目见 `references/rules-end-of-flow.md`。

跨步骤一致性审查是独立于单步骤品质自检的第二层审查，在所有步骤完成后执行。其目的是捕捉迭代过程中上游修改导致的下游不一致。

### 快速检查项目（所有模式通用）

| # | 检查维度 | 验证问题 |
|---|---------|---------|
| 1 | 目标用户一致性 | JTBD、定位、PR-FAQ 是否指向同一群人？ |
| 2 | 核心问题一致性 | PR-FAQ 是否针对 JTBD 陈述的问题？MVP 是否解决？ |
| 3 | 解法 ↔ 范围 | 选出的解法是否与 MVP 范围一致？ |
| 4 | 指标 ↔ 价值 | North Star 是否衡量 JTBD 成果？ |
| 5 | 风险时效性 | Pre-mortem 风险对最终解法是否仍然相关？ |

## 📐 设计原则

本文件遵循 Reviewer 设计模式：**检查标准与检查流程分开维护**。

- **流程不变**：步骤一（逐项自检）→ 步骤二（强制批判）→ 步骤三（格式呈现）
- **标准可换**：新增框架时，只需在「检查标准」段落新增对应清单
- **独立载入**：各 reference 文件中仍保留嵌入式清单作为内联提醒，本文件作为统一审查入口
