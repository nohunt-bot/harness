# 開発ハンドオフ — コアハンドオフパッケージ

> ユーザーが「開発を始めて」「開発ハンドオフパッケージを作成して」「プロジェクトをセットアップして」「Claude Codeに接続して」と言った時にトリガーされます。
> このファイルを読み込み、プロダクト企画プロセス全体の出力を統合し、Claude Code CLIで直接使用できる開発ハンドオフパッケージを生成してください。

## 環境制約とハンドオフ戦略

**重要な事実：Claude Chat / CoworkとClaude Codeは別のランタイム環境です — Chat内からClaude Codeを起動することはできません。**

ハンドオフ戦略は以下の通り：**構造化された開発ハンドオフパッケージ（ファイルセット）を作成**し、ユーザーがダウンロードしてプロジェクトフォルダに配置し、Claude Codeで単一のプロンプトで開発ワークフロー全体を開始できるようにします。

ハンドオフ方法はユーザーの環境によって異なります：

| ユーザー環境 | ハンドオフ方法 |
|-----------------|----------------|
| **Claude Chat（Web/App）** | ダウンロード用のzipファイルを作成；ユーザーがプロジェクトディレクトリに展開してClaude Codeを開く |
| **Claude Cowork（Desktop）** | 上記と同じ、ただしユーザー指定のローカルパスに直接ファイルを書き込める |
| **Claude Code内** | プロジェクトディレクトリにすべてのファイルを直接作成（このシナリオでは、このスキルはCLAUDE.mdから参照されている可能性が高い） |

## 開発ハンドオフパッケージ内容

以下のファイルセットを作成、すべてプロジェクトルートに配置：

```
[project-name]/
├── .gitignore             # バージョン管理除外（.env、シークレット、進捗ファイルなど — テンプレートはreferences/07c-architecture-setup.md参照）
├── CLAUDE.md              # Claude Codeのプロジェクトメモリ：プロダクトコンテキスト + 開発ガイドライン
├── TASKS.md               # 機能分解 + フェーズマイルストーン + タスクごとの受入基準
├── TICKETS.md             # チケット内容：タイトル、説明、チケットごとの受入基準 — PMが直接チケット作成可能
├── docs/
│   ├── PRD.md             # 完全なPRD（04-develop.md出力フォーマットから統合）
│   ├── ARCHITECTURE.md    # 技術アーキテクチャ：ディレクトリ構造 + DBスキーマ + APIエンドポイント + セキュリティアーキテクチャ
│   └── PRODUCT-SPEC.md    # プロダクトスペックサマリー（05-deliver.md → 4.6から統合）
└── scripts/
    └── setup.sh           # ワンクリック初期化スクリプト（ディレクトリ作成 + 依存関係インストール）
```

## 技術スタック確認フロー

ハンドオフパッケージを作成する前に、技術スタックを確認する必要があります。ユーザーが指定していない場合、以下の順序で質問：

### 必須質問（すべての出力に影響）

```
1. アプリケーションのタイプは？
   □ Webアプリ（ブラウザ）
   □ モバイルアプリ（iOS / Android / クロスプラットフォーム）
   □ デスクトップアプリ
   □ API / バックエンドサービス
   □ CLIツール
   □ その他

2. 希望の技術スタックはありますか？
   （ない場合、プロダクト特性に基づいて推奨します）
```

### 推奨ロジック（ユーザーが指定していない場合）

| アプリケーションタイプ | 推奨スタック | 根拠 |
|-----------------|-------------------|-----------|
| Webアプリ（高速MVP検証） | Next.js + Tailwind + Supabase | フルスタック一体、簡単デプロイ、Auth内蔵 |
| Webアプリ（複雑なバックエンドロジック） | React + Node.js/Express + PostgreSQL | 高い柔軟性、成熟したエコシステム |
| Webアプリ（Pythonチーム） | React + FastAPI/Django + PostgreSQL | Pythonエコシステム、DjangoはAdmin内蔵 |
| モバイルアプリ（クロスプラットフォーム） | React Native / Flutter | 単一コードベースで両プラットフォームカバー |
| APIサービス | FastAPI / Express / Go | 軽量、高パフォーマンス |

> Claudeは推奨時に根拠を説明し、ユーザーが推奨を上書きできることを伝えてください。

### オプション（プロダクトニーズに応じてフォローアップ）

```
3. ユーザー認証は必要ですか？（Authアプローチ選択に影響）
4. リアルタイム要件はありますか？（WebSocket / SSE）
5. ファイルアップロード/処理は必要ですか？（ストレージ選択に影響）
6. デプロイ先は？（Vercel / Railway / AWS / セルフホスト）
```
