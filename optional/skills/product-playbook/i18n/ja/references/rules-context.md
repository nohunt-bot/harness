# 📦 プロダクトコンテキスト蓄積ルール

> このファイルは`.product-context.md`のフォーマット、読み書きルール、シナリオ処理、コンフリクト解決を定義します。
> SKILL.md起動フローまたは各モードのS1プレステップにより読み込まれます。

## 1. ファイル位置＆ライフサイクル

- **パス**：プロジェクトルートディレクトリの`.product-context.md`（`.product-playbook-progress.md`と同じレベル）
- **永続保持**：このファイルはフロー終了時に**削除されません** — セッションを跨いで永続し蓄積されます
- **初回作成時**、ユーザーにリマインド：「⚠️ `.product-context.md`を`.gitignore`に追加することをお勧めします — このファイルには機密プロダクト戦略情報が含まれる可能性があります。」

---

## 3. 3シナリオ検出

起動時（進捗ファイルチェック後、モード選択前）に`.product-context.md`の状態を検出：

| 条件 | シナリオ | アクション |
|-----------|----------|--------|
| ファイルが存在し、`Core Strategy`セクションに実際の内容がある（空/プレースホルダーではない） | **シナリオ1：完全なコンテキスト** | サイレントに読み込み。表示：「📦 **[プロダクト名]**のプロダクトコンテキストを検出しました — 今回の企画セッションのベースラインとして使用します。」 |
| ファイルが存在しない | **シナリオ2：コンテキストなし** | この状態を記録。機能拡張またはリビジョンモード開始時にContext Bootstrapをトリガー（セクション4参照） |
| ファイルが存在し、`Core Strategy`は空またはプレースホルダーのみだが、`Decision History`に少なくとも1エントリある | **シナリオ3：部分的コンテキスト** | 既知情報のサマリーを表示し、補完オプションを提示（セクション5参照） |

**検出ロジック**：
1. ファイルは存在するか？
2. `Identity`セクションにProduct nameがあるか（プレースホルダーではなく）？
3. `Core Strategy`セクションにCore JTBDがあるか（プレースホルダーではなく）？→ はい = シナリオ1
4. `Decision History`セクションに`###`エントリがあるか？→ はいだが3がいいえ = シナリオ3

## 4. Context Bootstrap（シナリオ2のみ）

ユーザーが**機能拡張**または**リビジョンモード**に入るが`.product-context.md`がない場合、モードのS1の前に「ステップ0」を挿入。

**プレゼンテーション**：
```
📦 このプロジェクトでプロダクト企画ツールを使用するのは今回が初めてです。その後のフローをより効率的にするため、
まず基本的なプロダクト情報を収集します（約2-3分）。自動的に保存され、今後のセッションで利用できます。
```

### 段階的収集（すべての質問を一度にしない）

**ラウンド1（すべてのモードで必須）**：
- プロダクト名は？
- 一文で何をするか説明してください。
- プロダクトタイプは？（B2C / B2B / B2B2C / 社内ツール）

**ラウンド2（機能拡張で必須、リビジョンではオプション）**：
- 使用している技術スタックは？（言語、フレームワーク、データベース、インフラ）
- キーモジュールまたはサービスは？

**ラウンド3（リビジョンで必須、機能拡張ではオプション）**：
- DAU/MAUまたはリテンション率のデータはありますか？
- 最も一般的なユーザーフィードバックや不満は？
- 既知のセキュリティ問題や技術的負債はありますか？

### 技術スタック自動検出

ユーザーの口頭入力に加え、Bootstrapは**プロジェクトファイルを読み取って**検出を支援できます（読み取りのみ — ハードゲートには違反しない）：

| ファイル | 検出内容 |
|------|------------------|
| `package.json` | Node.jsエコシステム、フレームワーク、依存関係 |
| `Cargo.toml` | Rust |
| `go.mod` | Go |
| `requirements.txt` / `pyproject.toml` | Python |
| `Dockerfile` / `docker-compose.yml` | コンテナ化アーキテクチャ |
| プロジェクトルートディレクトリ構造（`src/`、`app/`、`lib/`など） | モジュール推論 |

検出後、**確認スタイル**で提示：
```
プロジェクトが使用しているものを検出しました：
- 技術スタック：Next.js 14 + TypeScript + PostgreSQL + Redis
- キーモジュール：auth/、billing/、dashboard/、api/
これで正しいですか？追加や修正はありますか？
```

ユーザーが確認した後にのみ`.product-context.md`に書き込み。

### Bootstrap完了後

収集した情報を`.product-context.md`に書き込み、未収集セクションは空のまま（プレースホルダーを使用）にし、モードの正式なS1に進む。

## 5. 部分的コンテキストの処理（シナリオ3）

`.product-context.md`が存在するがCore Strategyが空で、Decision Historyエントリのみがある場合：

**プレゼンテーション**：
```
📦 過去[N]回の企画セッションの記録があります：
- 技術スタック：[Decision Historyからマージした既知のスタック]
- 過去に修正したモジュール：[Decision Historyからマージした影響モジュール]
- コアプロダクト戦略はまだ記録されていません。

どうしますか？
  1️⃣ 直接開始（既知情報を使用、戦略セクションをスキップ）
  2️⃣ まず戦略情報を記入（JTBD、ポジショニング、North Star Metric）
  3️⃣ この情報は正しくない — 修正させてください
```

**自動再構築の試み**：すべてのDecision Historyエントリをスキャンし、`影響モジュール`、`スコープ`、`主要決定`から繰り返し登場するプロダクト名、技術スタック、モジュール名を抽出し、`Architecture & Tech Stack`に自動記入。`<!-- inferred from decision history -->`でマークし、推論元を示す。

## 6. コンテキスト自動読み込みルール

各モードのS1プレステップでコンテキストを読み込む際、**関連セクションのみを注入** — ファイル全体の内容をユーザーに表示しない：

| モード + ステップ | 注入コンテキストセクション |
|-------------|--------------------------|
| 機能拡張 S1 | Identity、Architecture & Tech Stack、最新3件のDecision Historyエントリ |
| リビジョン S1 | Identity、Core Strategy、Accumulated Insights（ペインポイント、PMF、セキュリティ）、最新3件のDecision Historyエントリ |
| Full/Quick/Build S1 | Identityのみ（プロダクト名、タイプ、ワンライナー） |
| 任意モードのPre-mortem | Security posture + Technical debt（Accumulated Insightsから） |

**肥大化制御**：Decision Historyはデフォルトで最新3件のエントリのみ注入。ユーザーは追加をリクエスト可能。

## 7. 空セクションスキップルール

コンテキストファイルが存在するが一部セクションが空の場合、**モード**に基づいてスキップまたは収集を判断：

| セクション | 機能拡張 | リビジョンモード | Full/Quick/Build |
|---------|------------------|---------------|-----------------|
| Identity | 必須（欠落時はBootstrap） | 必須（欠落時はBootstrap） | フロー自体がこれを生成 — 事前読み込み不要 |
| Core Strategy | **スキップ可能** | 必須（欠落時はS1内でクイックQ&A収集） | フロー自体がこれを生成 |
| Architecture & Tech Stack | 必須（欠落時はBootstrapまたは自動検出） | スキップ可能 | フロー自体がこれを生成 |
| Decision History | スキップ可能 | あれば含める、なければスキップ | フロー自体がこれを生成 |
| Accumulated Insights | スキップ可能 | あれば含める、なければスキップ | フロー自体がこれを生成 |

| 機能拡張 | Identity（確認のみ）、Architecture & Tech Stack（必須）、Core Strategy（スキップ可能） |

**原則**：空セクションは**フローをブロックしない**。現在のモードで「必須」のセクションが空の場合のみ、収集をトリガー。

## 8. コンテキスト自動書き込みルール（フロー終了時の抽出）

フロー終了時（`rules-end-of-flow.md`の終了条件チェックと同期して）、自動的にコンテキストを抽出：

| フロータイプ | 書き込み/更新セクション |
|-----------|-------------------------|
| Quick | Identity、Core Strategy（JTBD + North Star）、Decision Historyに追加 |
| Full | **すべてのセクション**（Identity/Strategy/Insightsを上書き、Historyに追加） |
| Revision | Core Strategyを更新（リポジショニングした場合）、Insightsを更新、Historyに追加 |
| Feature Extension | Architectureをマージ、Decision Historyに追加（機能固有テンプレート） |
| Custom | 完了したステップに対応するセクションを更新 |
| Build（7ステップ） | Identity、Core Strategy（部分的）、Historyに追加 |

### 書き込み戦略

| セクション | 戦略 | 備考 |
|---------|----------|-------|
| Identity | **最新で上書き** | 常に最新のフローのデータで上書き |
| Core Strategy | **最新で上書き** | 同上。リビジョン後の戦略がリビジョン前の戦略を置き換え |
| Architecture & Tech Stack | **マージ** | 新モジュールは追加、古いものは削除しない。新技術項目は追加 |
| Decision History | **追加のみ** | 過去のエントリを決して削除しない。フロー完了ごとに1エントリ追加 |
| Accumulated Insights | **マージ＆重複排除** | ペインポイントとフィードバックテーマは重複排除して追加。PMFとSecurityは最新値で上書き |

### Decision History追加テンプレート

**一般テンプレート**：
```markdown
### [ISO日付] - [フロータイプ]
- **スコープ**: [...]
- **主要決定**: [...]
- **特定されたリスク**: [...]
- **MVP境界**: [...]
- **成功メトリクス**: [...]
```

**機能拡張テンプレート**：
```markdown
### [ISO日付] - Feature Extension: [機能名]
- **問題**: [一文の問題ステートメント]
- **選択したソリューション**: [選択したソリューション + 根拠]
- **影響モジュール**: [影響モジュール]
- **スコープ**: [やること / 触らないこと]
- **受入基準**: [受入基準]
```

### 完了通知

```
✅ プロダクトコンテキストが`.product-context.md`に更新されました — 次回の企画セッションで自動的に読み込まれます。
```

## 9. コンフリクト解決

### ユーザーが既存コンテキストを修正

完全に許可。ユーザーが提供した修正は対応セクションを直接上書き（最新優先）。

### ユーザー提供データとコードベースのコンフリクト

S1プレステップで`.product-context.md`とプロジェクトファイル（例：`package.json`）の両方を読み取り、不一致が見つかった場合：

```
⚠️ 不一致が検出されました：
- コンテキストの記録：[コンテキストからの値]
- プロジェクトコードベース：[コードから検出された値]
どちらが正しいですか？
  1️⃣ コードベースを正とする（コンテキストを更新）
  2️⃣ コンテキストを正とする（移行中の可能性）
  3️⃣ 両方とも不完全 — 説明させてください
```

**処理原則**：
- **自動上書きしない** — ユーザーに判断を委ねる
- ユーザーの選択後に`.product-context.md`を更新
- 「移行中」が選択された場合、Architectureセクションに注記：`[移行中] React → Vue 3`
- コンフリクトをDecision Historyに記録

### フローからの新データがコンテキストを上書き

フロー中に生成されたデータがコンテキストの古いデータと異なる場合（例：リビジョンモードでJTBDを再定義）、**フローデータが優先**。フロー終了時に自動的に上書きされます。

## 10. 言語設定

`.product-context.md`が作成または更新される際、`Language Preference`セクションに言語設定を記録します：

- **インストール言語**：スキルインストールディレクトリの`.lang`ファイルから検出、またはユーザーのロケールから取得。
- **ユーザーの希望言語**：セッション中にユーザーがコミュニケーションに使用する言語。

**読み込みルール**：既存の`.product-context.md`を読み込む際、言語設定が記録されている場合、その言語でセッションを継続します。

**書き込みタイミング**：言語設定はContext Bootstrap（セクション4）時、またはコンテキストファイルを初めて作成するフロー終了時に書き込まれます。ユーザーがセッション中に明示的に言語を切り替えた場合に更新されます。
