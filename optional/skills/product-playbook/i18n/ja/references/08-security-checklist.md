# セキュリティチェックリスト

> 開発ハンドオフパッケージを作成する前に読み込みます。プロダクト企画フェーズで重要なセキュリティ要件が考慮され、セキュリティが後付けにならないことを保証します。

## 🔐 セキュリティアーキテクチャ クイックチェック

開発ハンドオフパッケージを作成する前に、以下の各セキュリティ側面を確認してください。各項目を✅（企画でカバー済み）または❌（追加が必要）で記入。

### 1. 認証＆認可

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| 認証方法が決定（JWT / Session / OAuth / Passkey） | | |
| トークンのストレージがセキュア（HttpOnly Cookie、localStorageではない） | | |
| トークンの有効期限とリフレッシュメカニズムが設計済み | | |
| パスワードストレージがbcrypt / argon2を使用（MD5/SHAではない） | | |
| 権限モデルが定義済み（RBAC / ABAC / シンプルロール） | | |
| すべてのAPIエンドポイントに対応する認可チェックあり | | |
| ログイン失敗にブルートフォース保護あり（ロックアウト / 段階的遅延） | | |
```

**JWTベストプラクティス（JWTを使用する場合）：**
- 短命のAccess Token（15-30分）+ 長命のRefresh Tokenを使用
- Refresh TokenをHttpOnly Secure Cookieに保存
- Token Revocationを実装（ログアウト時にRefresh Tokenを無効化）
- JWTペイロードに機密情報を保存しない

### 2. CORSポリシー（Cross-Origin Resource Sharing）

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| Allowed Originリストが定義済み（ワイルドカード*は不可） | | |
| 必要なHTTPメソッドのみ許可 | | |
| Access-Control-Allow-Credentialsが設定済み | | |
| Preflightキャッシュ期間が妥当（Access-Control-Max-Age） | | |
```

**CORS設定テンプレート：**
```
Allowed Origins：
  - 本番：https://[your-domain.com]
  - 開発：http://localhost:[port]

Allowed Methods：GET、POST、PUT、DELETE、PATCH
Allowed Headers：Content-Type、Authorization
Credentials：true（Cookie認証を使用する場合）
Max-Age：86400（24時間）
```

### 3. 入力バリデーション＆サニタイゼーション

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| すべてのAPI入力にサーバーサイドバリデーションあり | | |
| SQLインジェクション防止にパラメタライズドクエリを使用 | | |
| ユーザー入力のHTML出力前にエンコード（XSS防止） | | |
| ファイルアップロードにタイプ / サイズ制限あり | | |
| URL / リダイレクト先にホワイトリストバリデーション（Open Redirect防止） | | |
```

**バリデーション原則：**
- フロントエンドバリデーションはUX；バックエンドバリデーションはセキュリティ — 両方必要だが、バックエンドバリデーションは非交渉
- Schema Validation Library（例：Zod、Joi、Pydantic）で統一バリデーションロジックを使用
- 期待するフォーマットに一致しない入力はリジェクト — ユーザー入力を「修正」しようとしない

### 4. CSRF保護（Cross-Site Request Forgery）

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| 状態変更操作がPOST/PUT/DELETEを使用（GETではない） | | |
| CSRF Tokenを実装またはSameSite Cookieを使用 | | |
| 重要な操作に二次確認あり | | |
```

### 5. セキュリティヘッダー

```
| ヘッダー | 目的 | 推奨値 |
|--------|---------|-------------------|
| Content-Security-Policy (CSP) | XSS、データインジェクション防止 | default-src 'self'; script-src 'self' |
| X-Content-Type-Options | MIMEスニッフィング防止 | nosniff |
| X-Frame-Options | クリックジャッキング防止 | DENY または SAMEORIGIN |
| Strict-Transport-Security (HSTS) | HTTPS強制 | max-age=31536000; includeSubDomains |
| X-XSS-Protection | ブラウザXSSフィルター | 0（CSPに頼る方が信頼性が高い） |
| Referrer-Policy | リファラー情報の制御 | strict-origin-when-cross-origin |
| Permissions-Policy | ブラウザ機能の制限 | camera=(), microphone=(), geolocation=() |
```

### 6. APIセキュリティ＆Rate Limiting

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| APIにグローバルRate Limitingあり（例：100 req/min/IP） | | |
| 機密エンドポイントにより厳しい制限（ログイン 5 req/min、登録 3 req/min） | | |
| APIエラーレスポンスが内部詳細を漏洩しない（スタックトレース、SQL文） | | |
| APIバージョニング戦略が決定（/api/v1/） | | |
| 大量データエンドポイントにページネーション制限あり | | |
```

**Rate Limiting設計推奨：**
```
| エンドポイントタイプ | 推奨制限 | 識別方法 |
|--------------|-------------------|----------------------|
| 一般API | 100 req/min | IP + User ID |
| ログイン/登録 | 5 req/min | IP |
| パスワードリセット | 3 req/hour | IP + Email |
| ファイルアップロード | 10 req/min | User ID |
| 検索/クエリ | 30 req/min | IP + User ID |
```

### 7. アンチスクレイピング＆ボット対策

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| robots.txtが設定済み（機密パスを制限） | | |
| 重要フォームにボット保護あり（reCAPTCHA / hCaptcha / Honeypot） | | |
| APIにUser-Agentチェックあり（オプション） | | |
| 機密操作に行動分析あり（オプション、上級） | | |
```

**多層防御戦略：**
1. **基本層**：Rate Limiting + robots.txt — すべてのプロダクトが持つべき
2. **標準層**：+ CAPTCHA（登録/ログイン）+ Honeypotフィールド — B2Cプロダクトに推奨
3. **上級層**：+ 行動分析 + IP評判 + Device Fingerprint — 高リスクプロダクト

### 8. 機密データ保護

```
| チェック項目 | ステータス | 備考 |
|-----------|--------|-------|
| 機密データが転送中に暗号化（HTTPS/TLS） | | |
| 機密データが保存時に暗号化（要求される場合） | | |
| シークレットとキーがコードに保存されていない | | |
| .envと機密ファイルが.gitignoreに追加済み | | |
| ログにパスワード、トークン、クレジットカード番号を記録しない | | |
| 明確なデータ保持・削除ポリシー（該当する場合はGDPR） | | |
```

**シークレット管理推奨：**
- 開発：`.env`ファイル（バージョン管理外）+ `.env.example`（キー名のみ、値なし）
- 本番：プラットフォーム提供の環境変数管理を使用（Vercel Environment Variables / Railway Variables / AWS Secrets Manager）
- コミットメッセージ、PR説明、Issueでシークレットに言及しない


## 📎 統合タイミング

| トリガー | 統合アクション |
|---------|-------------------|
| 開発ハンドオフパッケージ作成前 | セキュリティクイックチェックを実行、結果をCLAUDE.md「リスクアラート」とARCHITECTURE.md「セキュリティアーキテクチャ」セクションに統合 |
| PRD作成時 | セキュリティチェック結果をPRD §6「技術的考慮事項 → セキュリティ要件」に統合 |
| Pre-mortemステップ | セキュリティ失敗シナリオの検討を促す |
| リビジョンモード S1 | 既存プロダクトの現在のセキュリティ態勢の提供を促す |

## 品質セルフチェック

```
| チェック項目 | ✅/❌ |
|-----------|------|
| 認証方法が明示的に選択され、「TBD」のままではない | |
| 少なくとも3つのセキュリティヘッダーが計画済み | |
| Rate Limiting戦略がプロダクト特性に合わせてカスタマイズ（テンプレートのコピーではない） | |
| .gitignoreがすべての機密ファイルを含む | |
| 「該当」とマークされたOWASP Top 10のすべての項目に対応措置あり | |
| セキュリティ対策の複雑さがプロダクトステージに合致（MVPに完璧なセキュリティは不要だが、基本は非交渉） | |
```
