## 2.2 April Dunfordのポジショニングフレームワーク

**対象：中/高完成度 / オーディエンスが経営層/セールス/マーケティングの場合**

ポジショニングはタグラインではありません — どこで誰と競争するかを決めることです：

```
| ポジショニング要素 | 質問 | あなたの回答 |
|---------------------|----------|-------------|
| 競合代替品 | あなたのプロダクトが存在しなかったら、ユーザーは何を使うか？（本当の答え。あなたが思う競合ではない） | |
| ユニーク属性 | 競合代替品にないものは何か？ | |
| ユーザーへの価値 | これらのユニーク属性がユーザーにもたらす具体的な価値は？ | |
| ターゲット市場特性 | この価値を最も重視するユーザーは誰か？（具体的であるほど良い） | |
| 市場カテゴリー | プロダクトを最適にポジショニングする市場フレームは？（フレームが競争基準を決める） | |
```

最もよくあるポジショニングの間違い：
- 機能をポジショニングとして扱う（「AIがあります！」はポジショニングではない）
- ポジショニングが広すぎ — 全員をカバーすると誰もカバーできない
- ユーザーの実際の代替品ではなく、自分が思う競合を基準にする

### 📝 ポジショニング品質チェックリスト
- ✅ 「競合代替品」はユーザーの視点からか？（ユーザーが実際に使っているもの、あなたが思う競合ではない）
- ✅ 「ユニーク属性」は競合代替品ができない、またはうまくできないことか？（双方が持っているものではない）
- ✅ 「ユーザーへの価値」はユーザーの言葉かプロダクトの言葉か？（「2時間節約」vs「AI搭載自動化」）
- ✅ 「ターゲット市場」は実際にその人々を見つけられるほど具体的か？
- ✅ 5つのポジショニング要素は論理的に一貫しているか？
- ❌ よくある問題：ユニーク属性と価値の断絶、間違った市場カテゴリーで間違った基準で判断される

## 2.3 HMW（How Might We）問題のリフレーミング

ペインポイントをHMW質問に変換し、JTBDの視点を組み合わせて各HMWの背後にあるジョブタイプを確認：

```
| ペインポイント# | ペインポイント | 対応JTBDタイプ | HMW質問 |
|---|---|---|---|
| P1 | [ペインポイントの説明] | 機能的 / 感情的 / 社会的 | どうすれば... |
| P2 | [ペインポイントの説明] | | どうすれば... |
```

HMWの粒度原則：
- 広すぎ（「ユーザーをもっと幸せにするには」）→ 方向性がない
- ちょうど良い（「ユーザーが初回セットアップを60秒で完了できるようにするには」）→ 制約がありつつオープン
- 狭すぎ（「ボタンの色を変えるには」）→ 可能性を制限

### 📝 HMW品質チェックリスト
- ✅ 明確な制約があるか？（完全にオープンエンドではない）
- ✅ 複数のソリューションの余地が十分にあるか？（単一の回答を指していない）
- ✅ JTBDまたはペインポイントに直接マッピングできるか？
- ✅ チームがこのHMWを見てソリューションのブレインストーミングを始められるか？
- ❌ よくある問題：広すぎ（ビジョンの再述）、狭すぎ（ソリューションを指定）、複数の問題を混在

**例：**
- ❌ 広すぎ：「どうすればユーザーの満足度を向上できるか？」
- ✅ ちょうど良い：「どうすれば初めてのマイホーム購入者が3分で手が届く住宅ローン額を算出できるか？」
- ❌ 狭すぎ：「どうすればホームページに住宅ローン計算機を追加できるか？」

## 2.4 機会評価表

HMW質問を優先順位付け：

```
| HMW質問 | 影響ペルソナ | ペルソナ規模 | ユーザーインパクト（1-5） | ビジネス価値（1-5） | 実現可能性（1-5） | 合計 | 優先度 |
|---|---|---|---|---|---|---|---|
| | [影響ペルソナをリスト] | [大/中/小] | | | | | |
```

**スコアリングスケールの定義：**

| スコア | ユーザーインパクト | ビジネス価値 | 実現可能性 |
|-------|-----------|---------------|-------------|
| 1 | 少数のユーザーにとって軽微な不便 | 間接的、長期的なリターンがせいぜい | 完全に新しい技術または大規模なR&Dが必要 |
| 2 | 一部のユーザーが時々遭遇 | 間接的にいくつかのメトリクスを動かす可能性 | 大幅な新規能力構築が必要（3ヶ月以上） |
| 3 | コアTAが定期的に遭遇 | 重要メトリクスへのポジティブな影響 | 新規開発が必要だが技術的に実現可能（1-3ヶ月） |
| 4 | 多くのユーザーが頻繁に遭遇 | ユーザー成長またはリテンションを直接ドライブ | 現在のチーム能力で2-4週間 |
| 5 | 多くのユーザーが日々コアタスクを完了できない | 収益またはNorth Star Metricに大きなインパクト | 現在のチームで2週間以内に完了可能 |

**Shreyas Doshiの機会コスト思考：**

「この機能のROIは？」ではなく、こう問いかけてください：

> 「Aにリソースを投資するなら、Bに投資する機会を放棄している。AはBより本当に価値があるか？」

ROI思考は単一の機会が追求する価値があるか評価する；機会コスト思考はすべての機会の中でより良い選択を助ける。

**0→1フォーカスリマインダー：** 機会評価を完了した後、MVPのコアとして**最優先のHMW質問1つだけ**を選ぶことを推奨します。（Facebook：大学生 → 高校生 → 全員；プロフィールページ → 写真 → ニュースフィード）
