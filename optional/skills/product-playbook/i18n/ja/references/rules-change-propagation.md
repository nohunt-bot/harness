# 🔄 変更伝播ルール

> ユーザーが完了済みステップを修正した場合に読み込みます。

## 📍 進捗インジケーター（各ステップで必ず表示）

いずれかのステップを実行する際、Claudeはレスポンスの一番最初に進捗バーを表示する必要があります。以下のフォーマットで：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 [実行モード] ｜ 進捗 S[現在のステップ番号] / S[総ステップ数]
✅ S1: [ステップ名]（完了）
✅ S2: [ステップ名]（完了）
▶️ S3: [ステップ名]（進行中）
⬜ S4: [ステップ名]（未着手）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

この進捗インジケーターは以下の場面で表示する必要があります：
- 新しいステップに入る時
- ユーザーがステップに戻って修正する時
- ステップを完了し、次のステップに進む前にユーザー確認を促す時

## トリガー方法
- 「ペルソナに戻る」「JTBDに戻る」「HMWに戻る」「PR-FAQに戻る」その他のステップ名
- 「[ステップ名]を修正したい」「[ステップ名] — 変更したい」
- 既に作成されたテーブルやコンテンツを直接参照して「これを変更して...」

## 修正後の必須アクション

いずれかのステップが修正された場合、Claudeは**以下のチェックを積極的に実行する必要があります**：

```
修正レイヤー               影響を受ける下流（再確認または更新が必要）
─────────────────────────────────────────────────
ペルソナ / JTBD            → HMW、機会評価表、ポジショニング、PR-FAQ、North Star、プロダクトスペックサマリー
HMW / 機会評価             → PR-FAQ、並行ソリューション、MVP、North Star、プロダクトスペックサマリー
ポジショニング             → PR-FAQ、プロダクトスペックサマリー
PR-FAQ / ソリューション    → Pre-mortem、GEM/RICE、MVP、Aha Moment、プロダクトスペックサマリー
MVP / Not Doing List       → User Story、DBスキーマ（既に生成済みの場合）、プロダクトスペックサマリー
North Star / メトリクス    → 仮説検証プラン、プロダクトスペックサマリー
プロダクトスペックサマリー → HTMLレポート、PRD（既に生成済みの場合）
```

### 機能拡張の依存関係：
```
機能拡張の依存関係：
─────────────────────────────────────────────────────
S1（問題 + コンテキスト）  → S2（ソリューション）、S3（リスク）、S4（実行スコープ）
S2（選択したソリューション）→ S3（リスク）、S4（実行スコープ）
S3（リスク評価）           → S4（実行スコープ）
```

## 実行プロセス

1. **影響範囲をユーザーに通知**：「[ステップ]を修正しました。これは[下流ステップのリスト]に影響します。それぞれ更新します。」
2. **下流項目を確認または自動更新**：
   - 下流の変更が軽微（文言の調整）→ 直接更新し、何が変わったか説明
   - 下流の変更が大幅（方向性の転換）→ 新しい方向性についてユーザーに確認を求めてから更新
3. **プロダクトスペックサマリーを再統合**
4. **HTMLレポートまたはPRDが既に生成されている場合**：直接再生成し、バージョンスナップショットを出力：

```
📋 バージョンスナップショット v[旧バージョン] → v[新バージョン]
修正ステップ：[ステップ名]
修正前の主要内容：[1-3文]
修正後の主要内容：[1-3文]
トリガーされた下流更新：[どのステップも調整されたか]
```

## 原則
- 修正がサイレントに行われてはならない — 影響範囲は必ず明示的に伝える
- ユーザーには「このステップのみ修正し、下流は今のところそのまま」を選ぶ権利がある。Claudeはどの部分が古くなっているかマーク（⚠️ 要更新ラベルを付加）
- 修正履歴は会話内で追跡可能に維持
