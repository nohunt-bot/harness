# 🔄 リビジョンモード ステップ順序（12ステップ + 最終出力）

> このファイルはリビジョンモードの正式なステップ定義です。SKILL.mdコアディスパッチャーにより読み込まれます。

## ステップ順序

```
Phase 0：現状分析
  S1.  既存プロダクトレビュー（ユーザーデータ概要 + コアメトリクス + 既知の問題 + セキュリティステータス）
  S2.  既存ユーザーJTBDの再検証（うまくいっているジョブは？いないジョブは？）

Phase 1：問題の収束
  S3.  ユーザーペインポイント収集（リテンション/チャーン分析 + ユーザーフィードバック統合 + 行動データ）
  S4.  ペインポイントサマリー表 → references/03-define.md → 2.1 を読み込み
  S5.  ポジショニング再評価 → references/03-define.md → 2.2 を読み込み（フォーカス：ポジショニング調整が必要か？）
  S6.  HMW質問リフレーミング → references/03-define.md → 2.3 を読み込み
  S7.  機会評価表 → references/03-define.md → 2.4 を読み込み

Phase 2：ソリューション設計
  S8.  PR-FAQ → references/04a-prfaq.md を読み込み（リビジョン後の体験を記述）
  S9.  Pre-mortem → references/04b-solutions.md → 3.3 を読み込み
  S10. MVPスコープ + Not Doing List → references/04c-mvp.md を読み込み（フォーカス：変更すること / 変更しないこと）

Phase 3：バリデーション
  S11. North Star + Aha Moment → references/05a-northstar-aha.md を読み込み（リビジョン前後のメトリクス比較）
  S12. 仮説検証プラン → references/05c-validation-spec.md を読み込み
────
最終出力 → プロダクトスペックサマリー（リビジョン版）
```

### S1 プレステップ：プロダクトコンテキスト読み込み

S1に入る前に、`references/rules-context.md`を読み込み、`.product-context.md`を確認：

- **完全なコンテキストあり（シナリオ1）**：PMFレベル、North Star、既知のペインポイント、セキュリティステータス、最新3件のDecision Historyエントリを自動反映。S1のガイダンスを**差分モード**に切り替え：「前回の評価では、PMFレベルは[X]、North Star Metricは[Y]でした。変化はありますか？最新のDAU/MAUとリテンション数値はどうですか？」— 以前収集した決定履歴と既知のペインポイントは再収集不要。
- **コンテキストなし（シナリオ2）**：Context Bootstrap（`rules-context.md`セクション4、ラウンド1 + 3）をトリガーし、以下の標準S1データ収集に進む。
- **部分的コンテキスト（シナリオ3）**：Decision Historyから機能変更履歴を読み込み（どのモジュールが変更され、どのリスクが特定されたかを把握）、全体的なプロダクト戦略とメトリクスについては確認（過去の作業は機能拡張のみで全体像が不足）。

### S1 標準ガイダンス

> リビジョンモードのS1はユーザーに既存プロダクトデータの提供を積極的に求めます：DAU/MAU、リテンション率、主要ユーザーフィードバック、過去のバージョン決定など。コンテキストが一部の回答を事前入力している場合は、再収集ではなく確認。
> S1では現在のセキュリティステータスも収集：既存の認証/認可メカニズム、既知のセキュリティ脆弱性や技術的負債、最近のセキュリティインシデント。この情報はリビジョンのリスク評価とPre-mortemに影響します。

### ファストパス

S1でユーザーが十分なデータ（ユーザーフィードバック、メトリクス、ペインポイントを含む）を提供した場合、S4-S7（ペインポイント → ポジショニング → HMW → 機会評価）を単一の会話ターンで作成可能、4回ではなく1回の確認のみ。トリガー条件：S3で収集したペインポイントリストに既に明確な優先順位付けとデータの裏付けがある。ハードゲートルールは変更なし — 各ステップの出力は完全に提示する必要がある；確認のリズムのみ加速。
