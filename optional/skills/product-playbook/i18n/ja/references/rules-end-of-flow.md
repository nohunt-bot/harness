# 🏁 エンドオブフロールール

> すべてのステップが完了した時に読み込みます。

## ⛔ 終了条件チェック

最終統合出力を作成する前に、以下を確認する必要があります：

1. 進捗インジケーターのすべてのステップが✅であることを確認
2. スキップされたステップがある場合（ユーザーの明示的リクエストによる）、最終出力で「⚠️ スキップ」とマーク
3. ⬜（未実行）のステップがある場合、最終出力に進まない
4. **セキュリティクイックチェック**：ユーザーが開発フェーズに入る場合（開発ハンドオフパッケージを生成）、最終出力にセキュリティリマインダーを含め、開発ハンドオフパッケージ生成時に自動的に`references/08-security-checklist.md`を読み込んで対応するセキュリティアーキテクチャセクションを作成

このルールの違反例：「残りのステップは重要ではない」と独自に判断してスキップする、未完了のステップを完了済みとマークする、複数のステップを単一の出力に統合する。

## 🔍 意思決定の一貫性チェック

最終統合出力を生成する前に、完了したすべてのステップをスキャンし、ステップ間の一貫性を検証します：

### モード別チェック項目

**🚀 クイックモード**（3ステップ — 2つの相互参照をチェック）：
1. JTBD ↔ PR-FAQ：PR-FAQはJTBDステートメントと同じ課題に取り組んでいるか？
2. PR-FAQ ↔ North Star：North Star MetricはPR-FAQが約束する成果を測定しているか？

**📦 フルモード**（20ステップ — 全7項目をチェック）：
1. ターゲットユーザー — JTBD、ポジショニング、PR-FAQ、North Starで同一のペルソナを参照しているか？
2. コア課題 / JTBD — PR-FAQは同じ課題に取り組んでいるか？MVPはそれを解決しているか？
3. ポジショニング — PR-FAQの見出しとソリューション方向に反映されているか？
4. ソリューション方向 — 選択されたソリューションはMVPスコープと一致しているか？
5. MVPスコープ — PR-FAQの約束と一致しているか？「やらないこと」の項目は遵守されているか？
6. North Star Metric — JTBDの成果を測定しているか？MVPスコープで達成可能か？
7. Pre-mortemリスク — 最終ソリューションとMVPを踏まえて依然として関連性があるか？

**🔄 リビジョンモード**（12ステップ — 4項目をチェック）：
1. 既存のJTBD ↔ ペインポイント ↔ ポジショニング：再評価後も一貫しているか？
2. PR-FAQ ↔ MVPスコープ：リビジョン範囲はPR-FAQの記述と一致しているか？
3. North Star ↔ Before/After：指標の比較は論理的か？
4. Pre-mortemリスク：改訂後のプロダクトに対して依然として関連性があるか？

**⚡ ビルドモード**（7ステップ — 4項目をチェック）：
1. 課題ステートメント ↔ PR-FAQ：同じ課題に取り組んでいるか？
2. ソリューション方向 ↔ MVPスコープ：選択されたソリューションはMVPと一致しているか？
3. North Star ↔ MVP：指標はMVPスコープで達成可能か？
4. Pre-mortemリスク：最終ソリューションに対して依然として関連性があるか？

**🔧 機能拡張モード**（4ステップ — 3項目をチェック）：
1. 課題 ↔ 選択したソリューション：ソリューションは述べられた課題を直接解決しているか？
2. ソリューション ↔ 実行スコープ：スコープは選択したソリューションを正しく実装しているか？
3. リスク評価：特定されたリスクは実行スコープに対して依然として関連性があるか？

**✏️ カスタムモード** — ユーザーが実際に実行したステップ間の相互参照のみをチェックします。カスタム選択に含まれなかったステップのチェックはスキップします。

### 実行方法
1. 各コア意思決定を1行のサマリーとしてリストアップ（例：「ターゲットユーザー：アーリーステージのスタートアップ創業者」）
2. ステップ間の矛盾や古くなった参照がないか確認
3. 不整合が見つかった場合：
   - 表示：「⚠️ 一貫性チェックにより最終出力前に [N] 件の問題が見つかりました：」
   - 各問題と影響を受けるステップをリストアップ
   - ユーザーに質問：「最終出力を生成する前にこれらを修正しますか、それともこのまま進めますか？」
4. すべて一貫している場合：
   - 表示：「✅ 意思決定の一貫性チェック合格 — すべてのステップが整合しています。」
   - 最終出力の生成に進む

### このチェックが重要な理由
反復的な企画プロセスにおいて、ユーザーは上流の意思決定（例：JTBDの変更）を修正することがありますが、一部の下流ステップには古いコンテンツが残ることがあります。このチェックは、最終ドキュメントが作成される前にそのギャップを検出し、一貫性のない成果物の発生を防ぎます。

## 📦 プロダクトコンテキスト自動抽出

すべてのステップが完了し最終統合出力を作成する間に、`references/rules-context.md`セクション8を読み込みコンテキスト抽出を実行：

1. **`.product-context.md`が存在するか確認**
   - 存在しない → 新規ファイルを作成
   - 存在する → ルールに従い更新（Identity/Core Strategyは上書き、Decision Historyは追加、Architectureはマージ、Insightsはマージ＆重複排除）

2. **コンテンツを抽出**（`rules-context.md`セクション8テーブルのフロータイプマッピングに従い）

3. **ユーザーに通知**：最終出力の後に表示：
   「✅ プロダクトコンテキストが`.product-context.md`に更新されました — 次回の企画セッションで自動的に読み込まれます。」

4. **バージョン管理リマインダー**（初回作成時のみ）：
   「⚠️ `.product-context.md`を`.gitignore`に追加することをお勧めします — このファイルには機密プロダクト戦略情報が含まれる可能性があります。」

### 出力言語オーバーライド

ユーザーは企画セッションとは異なる言語で出力をリクエストできます：
- 「PR-FAQを日本語で生成して」
- 「レポートをスペイン語で出力して」
- 「PRDを中国語で書いて」

言語オーバーライドがリクエストされた場合：
1. リクエストされた言語で出力コンテンツを生成する
2. フレームワーク名は英語のまま維持する（JTBD、PR-FAQ、North Star など）
3. 出力生成後は企画セッションの元の言語に戻る
4. 注意：これは出力ドキュメントの言語にのみ影響し、リファレンスファイルや企画フローには影響しない

## 拡張出力プロンプト

最終統合出力の完了後、積極的に質問：

```
「企画内容の統合が完了しました！以下のドキュメントを生成しますか？

□ 更新版 [ドキュメントタイプ]（アップロードされた元ドキュメントに基づくインクリメンタル更新）← 元ドキュメントがアップロードされた場合のみ表示
□ PDF ドキュメント（プロフェッショナルレイアウト、ブックマークナビゲーション付き、正式な共有に最適）
□ HTML企画レポート（インタラクティブ、オンライン共有に最適）
□ Word ドキュメント（共同編集に最適）
□ PRDエンジニアデリバリーパッケージ（フローチャート、DBスキーマ、UIワイヤーフレームを含む）
□ プレゼンテーション PPTX（会議プレゼンに最適、出力後 Keynote / PowerPoint での仕上げを推奨）
□ 開発ハンドオフパッケージ（CLAUDE.md + TASKS.md + TICKETS.md + 技術アーキテクチャ — Claude Codeですぐに開発開始可能）
□ 上記すべて

「不要です」と言って終了するか、特定のドキュメントを指定することもできます。
/export [pdf|docx|pptx|html|md] でいつでもエクスポートできます。」
```

**オプション表示ルール**：
- 元ドキュメントがアップロードされた → 「更新版 [ドキュメントタイプ]」を先頭に表示し「（推奨）」ラベルを付ける
- ターゲットオーディエンスがエンジニア → PRDと開発ハンドオフパッケージを先頭に
- ターゲットオーディエンスが経営層/リーダーシップ → PDFとプレゼンテーションを先頭に
- ターゲットオーディエンスがクロスファンクショナル → PDF、HTMLレポートとプレゼンテーションをすべて表示
- クイックモード → PDFまたはプレゼンテーションが必要かのみ確認
- ターゲットオーディエンスが自分自身 → 開発ハンドオフパッケージを先頭に

**エクスポートトリガールール**：
- ユーザーが PDF / Word / プレゼンテーション PPTX を選択 → `rules-export-document.md` を読み込み
- 初めてドキュメントエクスポートをトリガー → まず `rules-document-tools.md` を読み込み、必要なツールを確認・インストール
- ユーザーが HTML 企画レポートを選択 → `06-html-report.md`（既存ルール）を読み込み
- ユーザーが「上記すべて」を選択 → 各フォーマットを順次エクスポート
