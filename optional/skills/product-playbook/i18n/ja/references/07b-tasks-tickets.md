# 開発ハンドオフ — TASKS.md + TICKETS.md

## 📄 TASKS.mdテンプレート

機能分解のコア原則：
- MVP必須機能（P0機能）からスタート
- 各TaskはUser Storyにマッピング
- フェーズは明確な依存関係を持つ：Phase N+1はPhase Nの出力に依存
- 各Taskにはclaude Codeがセルフ検証可能な受入基準を含む

```markdown
# [プロダクト名] — 開発タスクリスト

## Phase 0：プロジェクト初期化
> 目標：実行可能な空のプロジェクトスケルトンを確立

- [ ] **T0.1** プロジェクト初期化（`scripts/setup.sh`またはマニュアル）
  - 受入基準：
    - [ ] `npm run dev` / `python manage.py runserver`または同等のコマンドが正常に起動
    - [ ] `.gitignore`が作成され、`.env`、`.env.local`、`node_modules/`、`.product-playbook-progress.md`、その他の機密ファイルを含む
    - [ ] `.env.example`が作成（キー名のみ、実際の値なし）
- [ ] **T0.2** Linter + Formatterの設定
  - 受入基準：lintがエラーなしでパス
- [ ] **T0.3** データベース設定 + 初回マイグレーション実行
  - 受入基準：データベースに接続可能、ベーステーブルが作成済み
- [ ] **T0.4** 基本ルーティング構造の設定
  - 受入基準：すべてのメインページルートがアクセス可能（空白ページでOK）

## Phase 1：コアフロー（Aha Momentへのパス）
> 目標：ユーザーがエントリーからAha Momentまでの最短パスを完了できるようにする
> 対応User Stories：[US-001、US-002、...]

- [ ] **T1.1** [機能名]
  - User Story：[ペルソナ]として、[アクション]をしたい、そうすれば[価値]
  - 受入基準：
    - [ ] [具体的なテスト可能な条件1]
    - [ ] [具体的なテスト可能な条件2]
  - 技術ノート：[必要なAPI / サードパーティサービス / 特殊ロジック]

- [ ] **T1.2** [機能名]
  - User Story：...
  - 受入基準：...

> **Phase 1完了チェックポイント**：ユーザーが[Aha Momentのアクション]を完了できる。できなければPhase 2に進まない。

## Phase 2：MVP完成
> 目標：Phase 1でカバーされなかった残りのP0機能をすべて実装
> 対応User Stories：[US-003、US-004、...]

- [ ] **T2.1** [機能名]
  - ...

> **Phase 2完了チェックポイント**：すべてのP0 User Storyの受入基準がパス。

## Phase 3：品質＆体験
> 目標：エラーハンドリング、エッジケース、ローディング状態、基本セキュリティ

- [ ] **T3.1** グローバルエラーハンドリング
- [ ] **T3.2** フォームバリデーション + エッジケース
- [ ] **T3.3** ローディング状態 + 空状態
- [ ] **T3.4** セキュリティチェック（`references/08-security-checklist.md`に従い各項目を確認）
  - 受入基準：
    - [ ] OWASP Top 10関連項目に対応（入力バリデーション、認証、XSS保護、CSRF保護）
    - [ ] セキュリティヘッダーが設定（CSP、X-Frame-Options、HSTSなど）
    - [ ] CORSポリシーが設定（ワイルドカード*は使用しない）
    - [ ] 機密APIエンドポイントにRate Limitingあり
    - [ ] APIエラーレスポンスが内部情報を漏洩しない
- [ ] **T3.5** レスポンシブデザイン（Webの場合）

## Phase 4：デプロイ
> 目標：外部ユーザーがアプリにアクセスできるようにする

- [ ] **T4.1** 環境変数管理
- [ ] **T4.2** デプロイ設定
- [ ] **T4.3** 基本モニタリング + ロギング
```

## 📄 TICKETS.mdテンプレート

TICKETS.mdはTASKS.mdの機能分解を取り、プロジェクト管理ツールでチケットを直接作成できる構造化コンテンツを作成します。各チケットにはPMが必要とするすべての情報が含まれます。

> **設計目標**：PMが各チケットの内容を直接Jira / Asana / Linearまたは他のツールにコピーしてチケットを作成できること。将来バージョンではAPIによる自動チケット作成をサポート予定。

> 生成日：[タイムスタンプ]

### チケットルール

1. **チケット-タスクマッピング**：TASKS.mdの各TaskはひとつのチケットにマッピNG（TKT-001 ↔ T0.1）；大きすぎるTaskは複数チケットに分割可能
2. **優先度の継承**：Phase 0-1はデフォルトP0、Phase 2はデフォルトP1、Phase 3-4はデフォルトP2 — RICEスコアに基づいて調整可能
3. **依存関係**：チケット間の依存関係を明示的にマーク、エンジニアがステップをスキップするのを防ぐ
4. **見積時間**：Task粒度原則（1-4時間）に基づき、合理的な見積りを提供
5. **推奨ラベル**：フェーズ、技術ドメイン（frontend / backend / database / infra）、機能モジュールを含む

### プロジェクト管理ツール連携（予約済み）

> 以下は将来の自動チケット作成のための予約インターフェース設計です。現在のバージョンでは手動PM用のTICKETS.mdのみを作成します。

|-------|-------------|--------------|----------------|

## 機能分解ロジック

MVP機能をTasksに変換するルール：

### フェーズ分割原則

```
Phase 0：プロジェクトスケルトン（すべてのモードで必須）
  → 初期化、linter、DB、基本ルーティング

Phase 1：Aha Momentまでの最短パス（最重要）
  → ユーザーエントリーからAha Momentまでに必要な最小限の機能
  → このパス上のP0機能のみ含む

Phase 2：MVP完成
  → Phase 1でカバーされなかった残りのP0機能
  → セカンダリフロー、サポートページ

Phase 3：品質＆体験
  → エラーハンドリング、エッジケース、ローディング/空状態
  → 基本セキュリティ、レスポンシブデザイン

Phase 4：デプロイ
  → 環境変数、デプロイ設定、モニタリング
```

### Task粒度原則

- 各Taskは**1-4時間**で完了可能であるべき
- 大きすぎる → サブTaskに分割（T1.1a、T1.1b）
- 小さすぎる → 関連Taskに統合
- 各Taskには少なくとも1つのテスト可能な受入基準が必要

### User Story → Taskマッピング

```
1つのUser Storyは1-3のTaskにマッピング可能：
  US-001：新規ユーザーとして、アカウントを登録したい、そうすればプロダクトを使い始められる
    → T1.1：登録ページUI
    → T1.2：登録API + データバリデーション
    → T1.3：メール認証フロー（MVPに必要な場合）
```
