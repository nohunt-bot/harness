# フェーズ4：Deliver — North Star + Aha Moment

## 4.1 Marty CaganのEmpowered Teams原則

**対象：高完成度 / 納品先オーディエンスがリーダーシップまたはクロスファンクショナルチームの場合**

> 4.1 Empowered Teamsは、納品先オーディエンスがリーダーシップまたはクロスファンクショナルチームの場合のみ表示；それ以外はスキップ。

```
| 次元 | Feature Team（回避すべき） | Empowered Team（目標） |
|-----------|---------------------|----------------------|
| 割り当て | 機能リスト（Output） | 解決すべき問題（Outcome） |
| 成功の定義 | 機能を期限通りにデリバリー | ユーザーとビジネスのメトリクスを達成 |
| PMの役割 | 要件収集者とプロジェクトマネージャー | 問題探索者とソリューション検証者 |
| エンジニアの役割 | 仕様を実装 | 問題探索とソリューション設計に参画 |
```

> "真のプロダクトディスカバリーは、エンジニアとデザイナーと**一緒に**行うものであり、PMが一人で完成品を渡すのではない。" — Marty Cagan

**Lennyの3つのPM責務：**
- **Shape**：ユーザーインサイト、データ、市場インテリジェンスを統合し、何を作るか決定
- **Ship**：高品質のプロダクトが予定通りリリースされ、サプライズがないことを保証
- **Synchronize**：すべてのステークホルダーがビジョン、戦略、目標、ロードマップで連携

## 4.2 成功メトリクスフレームワーク（North Star + 3層シグナル）

North Star Metricは以下を満たす必要があります：
- ユーザーが受け取る実際の価値を反映（バニティメトリクスではない）
- 継続的に成長可能（自然な上限がない）
- チーム全体を1つの目標で連携させる

```
| 企業 | North Star Metric | なぜこのメトリクスか |
|---------|-------------------|-----------------|
| Airbnb | 宿泊数 | ホストとゲスト双方に提供された価値を表す |
| Spotify | 月間リスニング時間 | ユーザーが本当に音楽を使い楽しんでいることを表す |
| Facebook | DAU / MAU比率 | 習慣的な再訪問を表す |
| Slack | 週間メッセージ送信数 | チームが本当にコラボレーションしていることを表す |
| Salesforce | アクティブ顧客ACV（年間契約額） | 顧客が継続的にビジネス価値を得ていることを表す（B2B） |
```

**あなたのNorth Star Metric：**
```
North Star Metric：[ユーザーとプロダクトのために創出されるコア価値を表す単一の数値]
定義：[正確な計算方法]
なぜこのメトリクスか：[実際のユーザー価値を表す理由を説明。ビジネス成果だけではなく]
```

### 📝 North Star品質チェックリスト
- ✅ ユーザーが受け取る価値を反映しているか？（売上やDAUではなく）
- ✅ 継続的に成長可能か？（自然な上限がない）
- ✅ チーム全員がこのメトリクスを見て何をすべきか分かるか？
- ✅ ゲーミング可能か？（可能なら、ガードレールメトリクスが必要）
- ✅ B2Bプロダクト：個々のユーザーだけでなく、組織レベルの価値を反映しているか？
- ❌ よくある問題：売上をNorth Starにする（売上は結果であり、ドライバーではない）、メトリクスが複合的すぎてアクションに結びつかない

**3層シグナルシステム（順番に達成する必要あり）：**

```
| レイヤー | メトリクスタイプ | 定義 | B2Cターゲット | B2Bターゲット |
|---|---|---|---|---|
| レイヤー1（前提条件） | コアアクション成功率 | ユーザーはプロダクトのコアアクションを完了したか？ | 30-40%以上 | 60-80%以上（ユーザーのモチベーションが高い） |
| レイヤー2（価値の代理指標） | D14 / D28リテンション率 | ユーザーは戻ってくるか？ | コンシューマープロダクト 15-20%以上 | ロゴリテンション 90%以上；Net Revenue Retention 100%以上 |
| レイヤー3（熱狂シグナル） | Sean Ellis Score | 「このプロダクトが使えなくなったら、どのくらいがっかりしますか？」 | 40%以上が「とてもがっかり」と回答 | 40%以上が「とてもがっかり」と回答 |
| ガードレールメトリクス | 過剰最適化の防止 | 他の重要な次元が損なわれないことを保証 | コンテキスト依存 | コンテキスト依存 |
```

注意：レイヤー1はレイヤー2の前提条件です。コアアクション成功率が非常に低い場合、リテンションデータは無意味です。ユーザーがプロダクトの価値を体験する機会がなかったからです。

## 4.4 Aha Momentデザイン

```
Aha Moment定義：
ユーザーが[特定の行動]を完了した時、このプロダクトのコア価値を体験したことになる。
目標：プロダクトに入ってから[X分 / Xステップ]以内にこの瞬間に到達させる。

Aha Moment到達率：[目標%]
現在の障壁：[Aha Momentにより早く到達することを妨げているものは？]
改善プラン：[障壁をどう取り除くか？]
```

**例：**
| プロダクト | Aha Moment | 時間目標 |
|---------|-----------|-------------|
| Slack | チームが2,000通目のメッセージを送信 | 最初の2週間 |
| Dropbox | 最初のファイルが2台目のデバイスに同期 | 初回使用から10分以内 |
| Zoom | 初回のワンクリック参加でスムーズな映像 | 初回使用時 |

### 📝 Aha Moment品質チェックリスト
- ✅ 具体的で追跡可能な行動か？（「プロダクトが便利だと感じる」ではない）
- ✅ JTBDの機能的ジョブに直接結びついているか？
- ✅ 時間目標は妥当か？（B2Cは初回使用内、B2Bはトライアル期間内が目安）
- ✅ オンボーディングを設計してより早く到達させることができるか？
