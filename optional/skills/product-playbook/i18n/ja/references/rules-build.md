# ⚡ ビルドモード ステップ順序（7ステップ + 最終出力）

> このファイルはビルドモードの正式なステップ定義です。SKILL.mdコアディスパッチャーにより読み込まれます。

> ⚠️ 必須リマインダー：「ユーザーリサーチフェーズをスキップすることは、ソリューションが仮定に基づいて構築されることを意味します。実行後できるだけ早くContinuous Discoveryを実施して検証することをお勧めします。」

## ステップ順序

```
S1. 問題ステートメントの確認（一文で）
S2. PR-FAQ → references/04a-prfaq.md を読み込み
S3. 並行ソリューション → references/04b-solutions.md → 3.2 を読み込み
S4. Pre-mortem → references/04b-solutions.md → 3.3 を読み込み
S5. GEM + RICE優先順位付け → references/04b-solutions.md → 3.4 + 3.5 を読み込み
S6. MVP + Not Doing List → references/04c-mvp.md を読み込み
S7. North Star + Aha Moment → references/05a-northstar-aha.md を読み込み
────
最終出力 → エンジニア向け実行サマリー
```

## 🔧 機能拡張クイックパス（4ステップ）

> ユーザーが**既存プロダクトに単一機能を追加**している場合、自動的にこのパスに切り替わります。
> トリガー条件：ユーザーの説明に「機能を追加」「新機能」「XX機能を追加」「既存システムに」「既存プロダクトに」などの表現が含まれる場合。

**フルビルドモード（7ステップ）との違い**：既存プロダクトにはすでにNorth Star、Aha Moment、プロダクトポジショニングがあるため、再定義の必要はありません。単一機能にはPR-FAQプレスリリースや完全なGEM+RICEの再優先順位付けは不要です。フォーカスは「何を追加するか、どう追加するか、既存機能が壊れないか」です。

### 機能拡張ステップ順序

```
S1. 問題 + 既存システムコンテキスト
S2. 3つの並行ソリューション + メリット/デメリット + AI推奨 → references/04b-solutions.md → 3.2 を読み込み
S3. リスク評価（リグレッション + 互換性） → references/04b-solutions.md → 3.3 を読み込み
S4. 実行スコープ（やること / 触らないこと / 受入基準） → references/04c-mvp.md を読み込み
────
最終出力 → 機能開発仕様
```

### S1 プレステップ：プロダクトコンテキスト読み込み

S1に入る前に、`references/rules-context.md`を読み込み、`.product-context.md`を確認：

- **完全なコンテキストあり（シナリオ1）**：プロダクト名、技術スタック、キーモジュール、最新3件のDecision Historyエントリーを自動で読み込み。S1のガイダンスを**確認スタイル**に変更：「プロダクトは[名前]で、[技術スタック]を使用し、キーモジュールには[モジュールリスト]があります。どの機能を追加しますか？影響を受けるモジュールは？」（質問2と質問3の一部が事前入力 — 確認のみ必要）
- **コンテキストなし（シナリオ2）**：Context Bootstrap（`rules-context.md`セクション4）をトリガーし、以下の標準S1に進む
- **部分的コンテキスト（シナリオ3）**：既知の技術スタックとモジュールを読み込み（Decision Historyからマージ）、不足部分を収集。例：「[既知のモジュール]以外に、影響を受ける可能性のあるモジュールはありますか？」

### S1 ガイダンス内容（問題 + 既存システムコンテキスト）

Claudeは以下の情報を収集する必要があります（段階的にガイド — すべての質問を一度にしない。コンテキストが事前入力した回答がある場合は、再収集ではなく確認）：

```
1. どの機能を追加したいですか？どんな問題を解決しますか？
2. 現在のプロダクトアーキテクチャ概要（技術スタック、キーモジュール）← コンテキストが事前入力可能
3. この機能はどの既存モジュールに影響しますか？← コンテキストが部分的に事前入力可能
4. この要件を裏付けるユーザーフィードバックやデータはありますか？
```

### インクリメンタルドキュメント出力（ソースドキュメントがある場合）

ユーザーがプロセス中にソースドキュメント（PRD、仕様書など）をアップロードした場合：

1. **インクリメンタル版**（ソースドキュメントがある場合のデフォルト）：
   - 元のドキュメント構造にセクションを挿入/修正
   - 元ファイルのフォーマット、スタイル、命名規則を維持
   - 新規コンテンツに `[NEW]` マークを付与
   - 変更コンテンツに `[UPDATED]` マークを付与し、元の内容を参考として保持
   - 新機能に関係のないセクションは一切変更しない

2. **スタンドアロン版**（ソースドキュメントがない場合）：
   - 標準の機能開発仕様フォーマットを使用（上記で定義済み）

3. **生成前にユーザーに確認**：
   「[ドキュメントタイプ]がアップロードされたことを検出しました。出力方法はどちらをご希望ですか？
    A) 元のドキュメントにインクリメンタル更新（推奨）
    B) スタンドアロンの機能開発仕様」
