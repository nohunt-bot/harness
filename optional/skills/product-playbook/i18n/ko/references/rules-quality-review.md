# 🔍 품질 리뷰 규칙

> 각 단계 완료 시 로드됩니다. 본 파일은 "리뷰 프로세스"와 "체크 기준"을 분리하여 관리하며, 품질 자체 점검 메커니즘이 실질적으로 작동하도록 보장합니다.

## ⚙️ 리뷰 프로세스 (모든 모드 공통)

각 단계의 산출물을 생성한 후, Claude는 다음 리뷰 프로세스를 실행해야 합니다:

### 1단계: 항목별 자체 점검

1. 해당 단계에 대응하는 품질 체크리스트를 확인 (아래 "체크 기준" 섹션 참조)
2. 각 항목을 ✅ 또는 ❌로 마크
3. ❌ 항목에는 "어떻게 개선할 것인가"에 대한 구체적인 설명을 반드시 포함

### 2단계: 강제 비판

- **모든 항목이 ✅가 되어서는 안 됨**: 모든 항목이 통과하면, "이 산출물의 가장 약한 부분"을 능동적으로 지적하고 강화 방법을 설명해야 함
- 이는 트집잡기가 아니라, 자체 리뷰가 형식적 승인이 되지 않도록 보장하는 것
- Amazon PR-FAQ의 정신 참고: 품질은 문제를 찾는 것이지, 문제가 없음을 확인하는 것이 아님

### 3단계: 표시 형식

```
📝 품질 자체 점검:
- ✅ [체크 항목]
- ✅ [체크 항목]
- ❌ [체크 항목] → 개선 방향: [구체적 설명]
⚠️ 가장 약한 부분: [설명] → 강화 제안: [구체적 행동]
```

## 📋 체크 기준 (단계별)

> 아래는 각 프레임워크 산출물의 독립적인 체크리스트입니다. 리뷰 프로세스는 변하지 않으며, 해당하는 체크리스트만 교체하면 됩니다.

### Persona (페르소나)

| # | 체크 항목 |
|---|----------|
| 1 | 세분화가 인구통계가 아닌 "용도/과업/동기"에 기반하는가? |
| 2 | 각 Persona가 MECE인가? (상호 배타적이며 타깃 시장을 완전히 커버) |
| 3 | 핵심 TA와 보조 TA가 명확히 구분되어 있는가? |
| 4 | 각 Persona의 "문제/과제"가 실제 관찰이나 합리적 추론에 기반하는가? |
| 5 | "현재 대처법과 이유"가 워크어라운드를 식별할 수 있을 만큼 구체적인가? |

흔한 문제: 나이/성별로 세분화, Persona 간 차이가 불명확, 페인포인트가 너무 모호함

### JTBD (Jobs to Be Done)

| # | 체크 항목 |
|---|----------|
| 1 | 구체적인 상황이 포함되어 있는가? ("언제 어디서나"가 아니라 "늦은 밤 은행에 연락할 수 없을 때") |
| 2 | 하나의 핵심 작업에 집중하는가? (세 개의 Job을 한 문장으로 합치지 않았는가) |
| 3 | 기능적/감정적/사회적 작업이 모두 식별되었는가? |
| 4 | "이 솔루션이 이 Job을 해결하는가"를 판단하는 데 사용할 수 있는가? |
| 5 | "현재 방법"과 "갭"이 포함되어 있는가? (갭 = 기회) |
| 6 | 5-Why 심층 탐구의 Q5가 감정적 동기/직업적 정체성/심리적 두려움에 도달했는가? (기능적 설명이 아닌) |

### Positioning (포지셔닝)

| # | 체크 항목 |
|---|----------|
| 1 | "경쟁 대안"이 사용자 관점에서 나온 것인가? (사용자가 실제로 대안으로 사용하는 것, 당신이 생각하는 경쟁사가 아닌) |
| 2 | "고유 속성"이 경쟁 대안이 할 수 없거나 잘 못하는 것인가? |
| 3 | "사용자에 대한 가치"가 사용자 언어인가 제품 언어인가? ("2시간 절약" vs "AI 기반 자동화") |
| 4 | "타깃 시장"이 이 사람들을 찾을 수 있을 만큼 구체적인가? |
| 5 | 다섯 가지 포지셔닝 요소 간에 논리적 일관성이 있는가? |

흔한 문제: 고유 속성과 가치 사이의 단절, 잘못된 시장 카테고리 선택으로 잘못된 기준으로 평가됨

### HMW (How Might We)

| # | 체크 항목 |
|---|----------|
| 1 | 명확한 제약 조건이 있는가? (완전히 개방적이지 않은) |
| 2 | 충분한 솔루션 공간이 남아 있는가? (유일한 해결책을 가리키지 않는) |
| 3 | 하나의 JTBD 또는 페인포인트에 직접 매핑되는가? |
| 4 | 팀이 이 HMW를 보고 솔루션 아이디어를 시작할 수 있는가? |

흔한 문제: 너무 큼 (비전 재진술과 동일), 너무 작음 (솔루션 지정과 동일), 여러 문제가 혼합됨

### PR-FAQ (보도자료 + FAQ)

| # | 체크 항목 |
|---|----------|
| 1 | 헤드라인이 사용자 관점에서 작성되었는가? ("사용자가 X를 할 수 있게 됨" vs "Y 기능 출시") |
| 2 | 첫 번째 문단의 처음 10초 이내에 독자가 "왜 이것이 중요한지" 이해할 수 있는가? |
| 3 | 페인포인트 설명이 실제 사용자 시나리오에서 나왔는가? |
| 4 | 솔루션 섹션의 첫 문장이 사용자 감정/시나리오로 시작하는가? (기능적 동사가 아닌) |
| 5 | 사용자 인용문이 실제 사람이 말할 법한 것인가? |
| 6 | FAQ에 기존 도구와의 비교에 관한 날카로운 질문이 포함되어 있는가? |

흔한 문제: 헤드라인이 뉴스가 아닌 제품 공지처럼 읽힘, 솔루션 섹션이 기능 목록이 됨, FAQ 질문이 모두 답하기 쉬움

### North Star Metric (북극성 지표)

| # | 체크 항목 |
|---|----------|
| 1 | 사용자가 받는 가치를 반영하는가? (매출이나 DAU가 아닌) |
| 2 | 지속적으로 성장할 수 있는가? (자연적으로 상한에 도달하지 않는) |
| 3 | 팀의 모든 사람이 이 지표를 보고 무엇을 해야 하는지 아는가? |
| 4 | 조작될 수 있는가? (그렇다면 가드레일 지표가 필요) |
| 5 | B2B 제품: 개인 사용자뿐만 아니라 고객 조직 수준의 가치를 반영하는가? |

흔한 문제: 매출을 North Star로 사용 (매출은 결과이지 추진 요인이 아님), 지표가 너무 복합적이어서 실행 불가

### Aha Moment (아하 모먼트)

| # | 체크 항목 |
|---|----------|
| 1 | 구체적이고 추적 가능한 행동인가? ("제품이 유용하다고 느낌"이 아닌) |
| 2 | JTBD의 기능적 작업과 직접 관련되는가? |
| 3 | 도달 시간 목표가 합리적인가? (B2C는 첫 사용 내, B2B는 트라이얼 기간 내) |
| 4 | 사용자가 더 빨리 도달하도록 온보딩을 설계할 수 있는가? |

단계 간 일관성 리뷰는 단일 단계 품질 자체 점검과 독립적인 2차 리뷰이며, 모든 단계 완료 후 실행됩니다. 그 목적은 반복 과정에서 상류 수정이 초래한 하류 불일치를 포착하는 것입니다.

## 📐 설계 원칙

본 파일은 Reviewer 디자인 패턴을 따릅니다: **체크 기준과 리뷰 프로세스를 분리하여 관리**.

- **프로세스는 불변**: 1단계 (항목별 자체 점검) → 2단계 (강제 비판) → 3단계 (표시 형식)
- **기준은 교체 가능**: 새로운 프레임워크 추가 시, "체크 기준" 섹션에 해당 체크리스트만 추가
- **독립 로딩**: 각 레퍼런스 파일에는 인라인 리마인더로 임베디드 체크리스트가 유지되며, 본 파일은 통합 리뷰 진입점으로 기능
