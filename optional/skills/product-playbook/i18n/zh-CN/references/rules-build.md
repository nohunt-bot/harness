# ⚡ 直接实作模式步骤序列（共 7 步 + 最终产出）

> 此文件为直接实作模式的权威步骤定义。由 SKILL.md 核心派发载入。

> ⚠️ 必须提醒：「跳过用户研究阶段，代表解法建立在假设上。建议执行后尽快进行 Continuous Discovery 验证。」

## 步骤序列

```
S1. 确认问题陈述（一句话）
S2. PR-FAQ → 读取 references/04a-prfaq.md
S3. 平行解法 → 读取 references/04b-solutions.md → 3.2
S4. Pre-mortem → 读取 references/04b-solutions.md → 3.3
S5. GEM + RICE 优先排序 → 读取 references/04b-solutions.md → 3.4 + 3.5
S6. MVP + Not Doing List → 读取 references/04c-mvp.md
S7. North Star + Aha Moment → 读取 references/05a-northstar-aha.md
────
最终产出 → 工程师导向执行摘要
```

## Reference 载入指示

| 步骤 | Reference 文件 |
|------|---------------|
| S1 | 无需外部 reference（直接引导使用者陈述问题） |
| S2 | `references/04a-prfaq.md` |
| S3-S5 | `references/04b-solutions.md` |
| S6 | `references/04c-mvp.md` |
| S7 | `references/05a-northstar-aha.md` |

## 最终产出格式

**工程师导向的执行摘要**：解法决策 → MVP 边界 → 成功指标 → 主要风险

完成后，依 `references/rules-end-of-flow.md` 执行流程结束规则。

## 🔧 功能扩充快速路径（4 步）

> 当侦测到使用者是在**既有产品上新增单一功能**时，自动切换到此路径。
> 触发条件：使用者描述中包含「加一个功能」「新增功能」「加入 XX 功能」「在现有系统上」「既有产品要加」等语句。

**与完整直接实作模式（7 步）的差异**：既有产品已有 North Star、Aha Moment、产品定位，不需要重新定义。单一功能不需要 PR-FAQ 新闻稿、GEM+RICE 全面重新排序。聚焦在「加什么、怎么加、会不会搞坏现有的东西」。

### 功能扩充步骤序列

```
S1. 问题 + 既有系统上下文
S2. 三平行解法 + 优缺点 + AI 推荐 → 读取 references/04b-solutions.md → 3.2
S3. 风险评估（回归 + 相容性）→ 读取 references/04b-solutions.md → 3.3
S4. 执行范围（做什么 / 不动什么 / 验收标准）→ 读取 references/04c-mvp.md
────
最终产出 → 功能开发规格
```

### S1 前置：产品上下文载入

进入 S1 前，读取 `references/rules-context.md` 并检查 `.product-context.md`：

- **有完整上下文（情境 1）**：自动带入产品名称、技术栈、主要模组、近 3 笔 Decision History。S1 引导改为**确认式**：「你的产品是 [名称]，使用 [技术栈]，主要模组包括 [模组清单]。这次要新增的功能是什么？会影响哪些模组？」（问题 2 和部分问题 3 已预填，只需确认）
- **无上下文（情境 2）**：触发 Context Bootstrap（`rules-context.md` Section 4），完成后再进入下方标准 S1
- **部分上下文（情境 3）**：带入已知的技术栈和模组（从 Decision History 合并），缺失部分仍需收集。例如：「除了 [已知模组]，还有其他会被影响的模组吗？」

### S1 引导内容（问题 + 既有系统上下文）

Claude 需收集以下资讯（逐步引导，不要一次丢出所有问题。若 context 已预填部分答案，改为确认而非重新收集）：

```
1. 要新增什么功能？解决什么问题？
2. 目前的产品架构概况（技术栈、主要模组）← context 可预填
3. 这个功能会影响哪些现有模组？← context 可部分预填
4. 有没有相关的用户反馈或数据支持这个需求？
```

|---------|---------|-----------|---------|

### S4 引导内容（执行范围）

```
**做什么（Scope）**：
- [具体要新增的功能项目]
- [需要修改的现有模组]

**不动什么（Do Not Touch）**：
- [明确不修改的模组和功能]
- [不动的原因]

**验收标准**：
- [ ] [可测试的具体条件]
- [ ] [回归测试：确认现有功能不受影响]
```

### 功能扩充最终产出格式

**功能开发规格**：问题陈述 → 选定解法 + 理由 → 影响范围 → 执行范围 + 验收标准 → 风险清单

### 增量文档产出（当有源文档时）

如果用户在流程中上传了源文档（PRD、规格书等）：

1. **增量版本**（有源文档时为默认）：
   - 在原始文档结构中插入/修改章节
   - 保持原始文件的格式、风格和命名规范
   - 新增内容标记为 `[NEW]`
   - 修改内容标记为 `[UPDATED]`，并保留原始内容作为参考
   - 与新功能无关的章节完全不动

2. **独立版本**（无源文档时）：
   - 使用标准的功能开发规格格式（如上方定义）

3. **产出前先询问用户**：
   「我检测到你上传了一份 [文档类型]。你希望产出方式是？
    A) 在原始文档上增量更新（推荐）
    B) 独立的功能开发规格」
